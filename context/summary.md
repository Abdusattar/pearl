3. **Найден и задокументирован реальный механизм бага «два имени — один

   вместо выбора из автокомплита — можно случайно попасть точным совпадением

   правило («жёлтая рамка — смотри подсказки, не печатай с нуля») уже
   покрывает это, просто не было соблюдено при тестировании.
4. **Подтверждено (важно для ожиданий): категоризация не задним числом.**

   никогда не пересчитывается автоматически, если потом поменять
   `product.expense_category_id`. Значит грумминг каталога чинит будущее
   сопоставление, но не переправляет уже проведённые суммы — для этого нужно
   вручную поправить и товар, и конкретную транзакцию. Счётчик «Без
   категории: N» на `/expenses/` уже считает именно транзакции (не карточки
   товаров) — правильная метрика для отслеживания, ничего строить не нужно.

   Не спроектировано и не реализовано — зафиксировать в

## Принятые решения

- **Пилот только на садике Сокулук (08.07):** Мунара, Абдусаттар, Махабат работают
  в системе только по объекту «Садик Сокулук», пока он не обкатан. Школу и садик
  Кожомкул сознательно не добавляем в систему до этого момента (сейчас их и так
  ещё нет в базе — см. «Цифры» ниже). После того как Сокулук обкатан — расширяться
  на остальные объекты.


- **PIN ребёнка:** формат 0001–9999 (4 цифры, никогда не переиспользуется — см. `app/services/students.py`). Сменили с 3 на 4 знака 03.07, пока родители ещё не пользовались боевым PIN — миграция была бесплатной. Причина: при 3 знаках и ~514 детей на всю организацию (школа+сады) запаса не хватило бы на пару лет текучки.
- **Долги/переплаты:** один общий баланс на ребёнка (не по видам услуг) — протокол банка не передаёт назначение платежа, родитель просто платит сумму. Учёба — индивидуальный тариф (`Student.extra.monthly_fee`), доп.услуги (проезд/секция) — справочник `Service` + чекбоксы на ребёнке (`StudentService`), начисляются вместе раз в месяц (`Charge`, генерируется при заходе на `/students/` или `/reports/`, без cron). **Ручной корректировки больше нет** (убрана 04.07 — см. ниже и [[architecture/billing_module]]).
- **Структура интерфейса (IA), обновлено 04.07:** Ученики = только личные данные + статус (PIN/ФИО/группа/Активен-Выбыл); карточка ребёнка (`/students/{id}/edit`) = ДВА независимых блока с раздельным сохранением — «Личные данные» и «Оплата», деньги/история сюда больше не входят; `/students/{id}/history` = баланс + единая лента начислений/оплат; `/reports/` = сводный долг/переплата по всем, фильтр по группе + «Только с долгом». Правило: у каждого вида данных — одно чёткое место, не дублировать по страницам. Подробная архитектура — [[architecture/billing_module]].
- **Просмотр/редактирование как режимы (04.07):** формы, которые правятся редко, но смотрятся часто (карточка ребёнка), открываются как текст, кнопка «Изменить» → редактируемые поля → «Сохранить» с подтверждением или «Отмена» без запроса на сервер. Причина: Айжан/Мунара нетехнические, боятся случайно испортить данные.
- **ФИО ребёнка (04.07):** разбито на `last_name`/`first_name` (обязательные) + `patronymic` (поле есть, не обязательно) — составное `name` сохраняется и пересчитывается на каждое сохранение для обратной совместимости (поиск, Optima, списки не трогали). Заодно добавлено `parent_name` (имя родителя, отдельно от телефона).
- **Банки:** Optima интеграция работает, банк сейчас тестирует. URL: `https://pearl-production-eef5.up.railway.app/optima/payment` — передан Аману/тех.специалисту, вместе с тестовыми PIN 9001/9002 (фиктивные аккаунты, не реальные дети)
- **Бакай API:** подтверждено Айжан (05.06) — QR оплаты, выписка, остаток
- **ИС стек:** FastAPI + PostgreSQL + HTMX/Jinja2 + OpenRouter Vision OCR + Docker → Railway
- **OCR:** Gemini 2.5 Flash Lite через OpenRouter
- **Авторизация:** bcrypt напрямую (не passlib) — passlib конфликтует с bcrypt на Windows
- **Пароли:** Абдусаттар=000, Айжан=001, Мунара=002, Махабат=003, Айдай=004, Талас=005 (установлены и в Railway DB, и локально — `scripts/add_user.py`)
- **org_id тестировщиков (12.07):** Абдусаттар/Мунара/Айдай/Махабат/Талас = 4 (Садик Сокулук, объект пилота), Айжан = 2 (Школа, реальная роль). owner/founder/director видят все организации — org_id только дефолтный контекст при входе, не ограничение доступа (`get_accessible_orgs()` в `app/dependencies.py`)

## Логика авто-склада (важно)

При подтверждении квитанции:
- Категория `warehouse_eligible = true` (только «Продукты питания») → показывается зелёный чекбокс «Добавить в склад»
- По умолчанию включён. При Провести → `WarehouseReceipt` создаётся для каждой позиции с qty и price
- Категория «Готовая еда» → `warehouse_eligible = false` → чекбокс не показывается → идёт прямо в расход
- `WarehouseReceipt.transaction_id` связывает складской приход с расходной транзакцией

## Локальная разработка

- **Docker не установлен на машине** (проверено 12.07, `docker`/Docker Desktop отсутствуют) — `docker-compose.yml` в репо нерабочий вариант для этой машины. Реально используется: Postgres как нативный Windows-сервис (`postgresql-x64-16`, `localhost:5432`, БД `pearl`) + `python -m uvicorn app.main:app --host 0.0.0.0 --port 8001` напрямую (зависимости стоят в системном Python, не venv).
- Postgres на этой Windows-машине уже поднят (`localhost:5432`, БД `pearl`) — можно гонять `uvicorn app.main:app --reload --port 8000` локально без Railway/Docker для быстрой проверки (порт 8000 или 8001 — оба использовались, не принципиально)
- **Готча (кириллица):** bash/curl в этой среде коверкает кириллицу — и на входе (`--data-urlencode` с русским текстом реально портит данные в БД), и **на выходе** (print() кириллицы из Python-скрипта в консоль показывает `�`, хотя сами данные корректны). Для HTTP-тестов с кириллицей — писать `.py`-скрипт с `requests`; для вывода результатов — писать в файл (`encoding="utf-8"`) и читать через Read, не через bash-консоль напрямую
- В dev-БД были рассинхронизированы sequence `organizations_id_seq`/`users_id_seq` (после ручного сидирования без `setval`) — почищено 04.07, но если где-то ещё заводили записи с явным id вручную, тот же класс бага может всплыть
- **Railway CLI установлен и залинкован** (`railway status`, `railway metrics`, `railway logs`) — можно диагностировать прод прямо из терминала без захода в dashboard. Подробности — [[architecture/deploy_ops]]

# Blueprint: Меню / приёмы пищи

> **Реализовано 10.07.2026** — актуальная архитектура теперь в
> [[architecture/menu_module]]. Этот файл — исторический черновик
> (проектирование ДО кода), оставлен как есть для истории решений с
> датами, дальше не обновляется.

## Зачем

Списание со склада на приём пищи (`WriteOff.meal_type`, готово с 09.07) не
привязано к конкретному блюду — нельзя посчитать себестоимость на ребёнка,
нет базы для будущего сравнения ожидание/факт (см. гап-анализ в
[[architecture/warehouse_module]]). Кроме того, планирование меню сейчас
нигде не фиксируется системно — оно либо в голове, либо на бумаге.

**Почему именно `dish_id` на списании (уточнено 10.07):** это конкретный
механизм, который делает возможной рецептуру снизу вверх — цепочка Меню
(блюдо на приём пищи) → Списание (продукты, привязанные к блюду через
`dish_id`) → Посещаемость (число детей на этот приём пищи) даёт «сколько
продукта уходит на одного ребёнка для конкретного блюда», посчитанное по
накопленным дням. Без `dish_id` списание привязано только к `meal_type`
(«обед» вообще), и продукты разных блюд одного обеда (суп/второе/компот)
было бы невозможно разделить между собой. Сам расчёт/отчёт по рецептуре —
отдельная будущая фича (см. «Не в V1»), но схема закладывается уже сейчас,
чтобы данные для него начали копиться с первого дня.

## Кто вводит (решено 10.07)

Тот же человек, что ведёт закуп и склад — в Сокулуке это **Махабат**
(`role=staff`). Осознанное решение: в садике мало персонала, не плодить
отдельную роль под меню. Доступ — тот же, что уже есть у неё к складу.

## Модель данных

### Dish (новая таблица, каталог блюд)
```
id, name (unique), created_at
```
Глобальный каталог (не org-scoped) — по аналогии с `Product`. Простой
список названий, без состава/граммовок в V1 (см. «Не в V1» ниже).

**Защита от опечаток (уточнено 10.07):** создание блюда идёт через
`get_or_create_dish()` — та же логика, что уже есть в
`app/services/products.py` (`get_or_create_product`): точное совпадение →
затем нечёткий поиск (rapidfuzz) среди существующих блюд, порог схожести
как у товаров → и только если ничего похожего не нашлось, создаётся новая
запись. Без этого «Каша рисовая» и «Каша ристовая» стали бы двумя разными
блюдами и разбили бы будущую статистику по рецептуре на два несвязанных
ряда данных. Отдельная таблица алиасов (`DishAlias`, по аналогии с
`ProductAlias`) в V1 не нужна — каталог блюд ожидается маленьким
(десятки, не сотни), фаззи-поиска по имени достаточно.

### MenuEntry (новая таблица)
```
id, organization_id, date, meal_type, dish_id (FK Dish),
created_by, created_at
```
`meal_type` — те же значения, что уже использует `WriteOff.meal_type`.
**Подтверждено Мунарой (10.07): 4 приёма пищи — завтрак, обед, полдник,
ужин** (полный день садика, не 3 как предполагалось раньше). **Без
уникального ограничения на
(organization_id, date, meal_type)** — на один приём пищи может быть
несколько блюд, каждое отдельной строкой. **Решено (10.07): обед —
несколько блюд** (суп, второе, компот отдельно), не один пункт.

### WriteOff (расширение существующей таблицы)
```
+ dish_id  Integer, ForeignKey("dishes.id"), nullable=True
```
Nullable осознанно — списание должно проходить и без выбранного блюда (см.
решение по блокировке ниже).

## Экраны и роли

- **`/menu/`** — таблица на неделю (дни × приёмы пищи), доступна роли
  `staff` и выше. Ячейка — мультивход блюд с автокомплитом по каталогу
  `Dish` + inline-создание нового блюда (тот же UX-паттерн, что уже
  используется для товаров на актуализации склада — fuzzy-поиск,
  создание на лету). Можно заполнить на 1 неделю (минимум) или сразу на 2.
- **`/warehouse/writeoff/meal`** (существует) — дополняется выбором блюда:
  dropdown блюд из `MenuEntry` этой даты/приёма пищи. Если меню на эту
  дату не заполнено — dropdown пустой, но форма **не блокируется** (см.
  решение ниже), списание проходит с `dish_id = NULL`.
- **Предупреждение** — баннер при заходе в систему (или на `/menu/`),
  если у ближайшего будущего буднего дня нет ни одной `MenuEntry`.
  Не блокирует ничего, просто напоминание — тот же принцип, что уже
  применён в актуализации склада («подсвечивать, не блокировать»).

## Правило по срокам (решено 10.07)

Меню на понедельник должно быть готово не позже пятницы предыдущей недели
— то есть минимум на несколько дней вперёд, не в последний момент. Это
организационное правило (проговорить с Мунарой/Махабат), в системе
выражается только через предупреждающий баннер, не через хард-блок.

## Решение: блок vs предупреждение (10.07)

Если на дату/приём пищи ещё нет меню, а списание уже делают — **система
мягко предупреждает, но разрешает списать**. Согласуется с уже принятым в
проекте принципом (актуализация склада, расхождение ожидание/факт — везде
подсветка, не блокировка). Жёсткая блокировка была рассмотрена и
отклонена — риск застопорить реальную работу повара в моменте (забыли
внести меню — обед всё равно нужно готовить и списывать).

## Расписание сада (уточнено 10.07)

Сокулук работает Пн–Пт, без выходных выхода на работу, без сезонной
разницы (лето/зима) в структуре приёмов пищи — не нужно закладывать
варианты по дням недели или сезону в модель данных.

## Ключевые сценарии

1. **Happy path**: в пятницу Махабат заходит на `/menu/`, видит таблицу на
   следующую неделю (Пн–Пт), заполняет блюда по приёмам пищи через
   автокомплит, сохраняет.
2. **Забыли заполнить**: наступил день, а меню не внесено — Махабат видит
   предупреждение, при этом списание на приём пищи всё равно возможно без
   блюда (`dish_id = NULL`). Задним числом привязать такой `WriteOff` к
   блюду после того как меню внесли — **не в V1** (см. ниже).
3. **Новое блюдо**: блюда нет в каталоге — вписывается вручную, создаётся
   новая запись `Dish` на лету (как с товарами).

## Явно НЕ в V1

- Состав/граммовки блюда (норма продуктов на порцию)
- Прогноз ожидаемого расхода продуктов из меню (сравнение ожидание/факт)
- UI для родителей / бота, показывающий меню
- Задним числом привязка уже созданных `WriteOff` к блюду
- Копирование меню с прошлой недели одной кнопкой (может появиться позже
  как удобство, не гарантировано в V1)
- Разные варианты меню по сезону/дню недели (не нужно — сад работает
  одинаково Пн–Пт круглый год)

Будущее (после накопления истории): предлагать состав блюда по типовой
раскладке (например, из готовой базы рецептов), Мунара/Махабат подтверждают
или правят вручную — не автоматизировать без подтверждения человеком.

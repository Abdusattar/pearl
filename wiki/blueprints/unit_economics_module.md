## Зачем

Себестоимость на ребёнка (юнит-экономика) требует ВСЕХ потоков расходов
вместе, не только питания/хозтоваров (см. `project_brief.md`, «Стратегическое
видение — продолжение 08.07»). Собрали 10.07 полную картину того, чего не
хватает — см. `logs/terminal/2026-07-10.md`.

## Стратегическое решение 10.07 — пересмотр архитектуры Pearl vs 1С

Решение «закрыто 18.06» (Зарплата/Амортизация — целиком зона 1С) **частично
пересмотрено**: Pearl не должен структурно зависеть от наличия 1С у клиента —
продукт метится в будущем как единый софт для садиков, а не все садики имеют
1С. Решение НЕ означает, что Pearl берёт на себя официальный бухучёт (НДФЛ,
Соцфонд, проводки, налоговая отчётность) — это по-прежнему всегда зона
бухгалтера/1С-аналога, везде, где он есть. Pearl считает **управленческую
оценку «в достаточном виде»** — для внутренней картины расходов и
юнит-экономики, не для официальной отчётности.

## Сотрудники / ФОТ

### Employee (новая таблица)

Активация модуля, который уже был в изначальном списке `is_modules.md`
(«Сотрудники: карточка, роль, зарплата») — не новый скоуп.

```
id, organization_id, full_name, role, salary (оклад),
status (active/уволен, по аналогии со Student — active/archived),
created_by, created_at
```

**Решено 10.07: полноценный реестр** (не одна общая сумма) — на каждого
сотрудника отдельная карточка с окладом, не агрегат.

**Статус — отметка, не удаление (решено 10.07).** Тот же паттерн, что уже
есть у `Student` (Активен/Выбыл, история не теряется) — увольнение переводит
в статус «Уволен», запись остаётся для отчётности задним числом. Отпуск
отдельного статуса не требует — зарплата в отпуске всё равно платится, на
ФОТ не влияет, только увольнение убирает сотрудника из суммы.

### ФОТ считается на лету — НЕ хранится как Transaction (пересмотрено 10.07)

**Исходный план (генерировать `Transaction` тем же паттерном, что
`generate_monthly_charges()`) отклонён после разбора с эдвайзером** —
паттерн начислений безопасен только потому, что список детей уже заполнен
на момент первого захода. Список сотрудников стартует **пустым** и
заполняется постепенно — при таком же паттерне первый заход на
`/employees/` (когда сотрудников ещё 0) заморозил бы ФОТ текущего месяца на
0 навсегда (идемпотентность не даёт перегенерировать). Та же проблема при
любом увольнении/найме/смене оклада в течение месяца — начисление один раз
в месяц просто не подходит для величины, которая меняется чаще.

**Решение:** ФОТ — не строка в `Transaction`, а **вычисляемая на лету
величина**: `SUM(Employee.salary WHERE status=active AND organization_id=X)`.
Пересчитывается каждый раз, когда нужна цифра — не хранится, не
«замораживается», не идемпотентна и не нуждается в этом. Показывается на
`/employees/` (итог сверху) и в будущем — в отчёте по юнит-экономике.
Не попадает в `/expenses/`/Закуп вообще — это не покупка, у неё нет
поставщика/чека, попадание в тот список исказило бы счётчики долга
поставщику и «без категории», рассчитанные на реальные закупки.
Категория `ExpenseCategory` «Зарплата» в итоге **не нужна** — ФОТ никогда
не становится строкой расхода.

## Амортизация

### Asset — расширение существующей таблицы

```
+ useful_life_months  Integer, nullable
```
Nullable — не все активы обязаны амортизироваться (например, актив уже
полностью учтён иначе, или это МБП, не ОС). Если `NULL` — не участвует в
расчёте.

### Расчёт — чистая функция от даты, тоже НЕ Transaction (пересмотрено 10.07)

```
monthly_amount(asset) = cost / useful_life_months
depreciated_to_date(asset, on_date) = min(cost, monthly_amount × months_since(purchase_date, on_date))
сумма месяца = depreciated_to_date(конец месяца) − depreciated_to_date(начало месяца)
```
Автоматически обнуляется, когда актив полностью самортизирован — без
отдельного флага/поля состояния.

**Тем же эдвайзер-разбором отклонён вариант «генерировать Transaction раз в
месяц»** — амортизация не «текущая сумма» вроде ФОТ, а ряд по времени
(стартует через месяц после покупки, заканчивается когда актив
самортизирован); при попытке хранить это как ежемесячные `Transaction`
понадобился бы докат пропущенных месяцев для старых активов и отдельная
логика остановки. **Чистая функция снимает это полностью** — считается
заново на любую дату по `Asset.cost`/`purchase_date`/`useful_life_months`,
без хранения истории и без докатки задним числом. Показывается на
`/assets/` (итог по месяцу сверху списка), в `/expenses/`/Закуп не попадает
— по тем же причинам, что и ФОТ (не покупка, нет поставщика/чека).
Категория `ExpenseCategory` «Амортизация» тоже не нужна.

**Начисление стартует со следующего месяца после покупки** (не с месяца
самой покупки) — уточнено финэкспертом 10.07, стандартная практика.
`months_since()` в формуле выше считает от начала следующего месяца после
`purchase_date`, не от самой даты.

### Сроки полезного использования по умолчанию (ответ финэксперта, 10.07)

| Категория актива | Срок | Группа НК КР |
|---|---|---|
| Мебель (общая) | 5 лет (60 мес.) | Группа 2 (25%) |
| Игровые комплексы | 5 лет (60 мес.) | Группа 3 (20%) |
| Кухонное оборудование | 5 лет (60 мес.) | Группа 2 (25%) |
| Мелкий инвентарь/игрушки | 2 года (24 мес.) | Группа 1 (30%) |
| Бытовая техника/ТВ | 3 года (36 мес.) | Группа 2 (25%) |

Используются как значение по умолчанию при создании актива (по
`Asset.category`), с возможностью поправить вручную на конкретном активе.
Полная (первоначальная) стоимость берётся для расчёта, без ликвидационной
стоимости.

**Нестыковка 50к/10к — разрешена (10.07).** Это два разных порога, не
противоречие:
- **< 10 000 сом** (НК КР ст.225 п.1,5) — налоговый порог, ниже него предмет
  вообще не признаётся амортизируемым ОС для налогового учёта → расход сразу
- **10 000–50 000 сом** — зона МБП/Инвентарь (складская модель, как
  хозтовары) — списывается при выдаче, не капитализируется
- **> 50 000 сом + служит > 1 года** — Основное средство (капитализация +
  амортизация). Порог из практики бюджетных учреждений КР (Приказ МФ №1-п),
  для Жемчужины как частной школы не обязателен по закону, но рекомендован
  экспертом как критерий существенности — иначе пришлось бы заводить карточку
  ОС на каждый недорогой стул

Полная сводная таблица — [[architecture/expense_categories]]. Порог
применяется не только к игрушкам (как было сформулировано 09.07), а ко всем
непищевым закупкам в принципе.

**Это управленческая оценка, не официальная проводка** — не противоречит
ответу финэксперта 09.07 («расчёт амортизации = 1С»), тот ответ был про
официальный бухучёт (счёт 7150), это разные вещи с разным назначением.

## Не в V1

- НДФЛ/Соцфонд/удержания — никогда не в Pearl, это зона 1С-аналога
- История изменений оклада (кто сколько получал в прошлом) — V1 хранит
  только текущий оклад, не версионирует. Раз ФОТ считается на лету (не
  хранится), это означает: ФОТ «на сегодня» всегда точный, но ФОТ «на
  произвольную дату в прошлом» посчитать нельзя (нет истории окладов) —
  приемлемо для V1, отчёт по прошлым периодам не в скоупе
- Тарифный лимит на вычет рекламных расходов по НК КР — уточнить у
  финэксперта отдельным вопросом (см. `finance_expert_prompt_categories.md`),
  не блокирует реализацию категории

## Доступ к окладам — решено 10.07

Новая роль **`founder`** — не переиспользуем `role=owner` (тот уже занят
Абдусаттаром как техническим/супер-админским доступом, `seed.py` id=1).
Заводим Айдай (реальный бизнес-владелец) отдельной ролью.

Затрагивает `app/dependencies.py`:
- `get_accessible_orgs()` — `founder` добавляется в ту же группу, что
  `("owner", "director")` — полный доступ ко всем организациям (школа +
  оба садика), founder должен видеть всю финансовую картину, не по одной
  локации
- Редактирование `Employee.salary` — доступно `role in ("owner", "founder")`,
  не Мунаре/Махабат/Айжан (чувствительные данные о зарплатах — уже отдельно
  от их обычного доступа к складу/детям/расходам)

# Модуль Меню/приёмы пищи — Архитектура и решения

> Реализовано: 10.07.2026. Blueprint (проектирование ДО кода) —
> [[blueprints/menu_module]], там же полная история решений с датами.

## Зачем

Списание со склада на приём пищи (`WriteOff.meal_type`, готово с 09.07) не
было привязано к конкретному блюду — нельзя было посчитать себестоимость на
ребёнка, не было базы для будущего сравнения ожидание/факт (см. гап-анализ
в [[architecture/warehouse_module]]).

**Почему `dish_id` на списании:** цепочка Меню (блюдо на приём пищи) →
Списание (продукты, привязанные к блюду) → Посещаемость (число детей) даёт
«сколько продукта уходит на одного ребёнка для конкретного блюда» —
рецептура снизу вверх, из фактических данных, без ввода норм заранее. Сам
расчёт/отчёт по рецептуре — будущая фича, схема закладывалась уже сейчас,
чтобы данные начали копиться с первого дня.

## Модель данных

### Dish
```
id, name (unique), created_at
```
Глобальный каталог (не org-scoped), как `Product`. Только название, без
состава/граммовок.

### MenuEntry
```
id, organization_id, date, meal_type, dish_id (FK Dish),
created_by, created_at
```
4 приёма пищи — Завтрак/Обед/Полдник/Ужин (подтверждено Мунарой). Без
уникального ограничения на `(organization_id, date, meal_type)` — обед
обычно несколько блюд (суп/второе/компот), каждое отдельной строкой.

### WriteOff.dish_id (расширение)
```
+ dish_id  Integer, ForeignKey("dishes.id"), nullable=True
```
Nullable осознанно — списание проходит и без выбранного блюда (мягкое
предупреждение, не блок, см. ниже).

Миграция: `d4e5f6g7h8i9`.

## Защита от опечаток

`get_or_create_dish()` (`app/services/dishes.py`) — та же логика, что у
товаров (`get_or_create_product`): точное совпадение → нечёткий поиск
(rapidfuzz, порог 85) среди существующих блюд → и только если ничего
похожего не нашлось, создаётся новая запись. Без `DishAlias` — каталог
блюд маленький, фаззи-поиска по имени достаточно.

## Экраны

### `/menu/` — карточки по дням

Не широкая таблица (плохо ложится на телефон), а **карточка на каждый
день** (Пн–Пт), внутри — 4 блока по приёмам пищи. В каждом блоке:

1. **Чипы «обычно даём»** — блюда, чаще всего встречавшиеся в `MenuEntry`
   для этого `meal_type` (`frequent_dishes()`, топ-8 по частоте). Клик —
   добавляет без печатания. Формируется само по мере накопления истории —
   первую неделю почти пусто, дальше наполняется.
2. **Уже выбранные блюда** — чипы с крестиком для удаления.
3. **Поле с автокомплитом** — печатать новое блюдо, Enter добавляет чип
   (или клик по подсказке из `/menu/dishes/search`).

`GET /menu/?org_id=&start=&weeks=` — по умолчанию открывает ближайший
понедельник, `weeks` (1–4) расширяет диапазон, кнопка «+ Добавить ещё
неделю» просто увеличивает `weeks` в URL. `POST /menu/` пересохраняет весь
диапазон целиком (удаляет старые `MenuEntry` за эти даты, вставляет
заново по пришедшим тройкам `entry_date`/`entry_meal`/`entry_dish`) —
проще и предсказуемее частичного upsert.

**Предупреждение, не блок:** баннер наверху страницы, если у ближайшего
будущего буднего дня нет ни одной `MenuEntry`. Тот же принцип, что уже
применён в актуализации склада.

### `/warehouse/writeoff/meal` — выбор блюда на списании

Каждая строка товара получила необязательный `<select>` «Блюдо» —
подгружается через `GET /menu/dishes/for-meal?org_id=&date=&meal_type=`
при смене даты/приёма пищи вверху формы. Если меню на эту дату не
заполнено — список пуст, показывается предупреждение
(`#no-menu-hint`), но списание **не блокируется** — `dish_id` остаётся
`NULL`.

## Кто вводит

Тот же человек, что ведёт закуп/склад — в Сокулуке это **Махабат**
(`role=staff`), не отдельная роль. Меню на понедельник должно быть готово
не позже пятницы предыдущей недели (организационное правило, не
enforced в коде).

## Баг: чипы-подсказки не кликались (найден 15.07, подтверждён и задеплоен 16.07)

`onclick="addChip(...,{{ d.name|tojson }})"` внутри HTML-атрибута,
ограниченного двойными кавычками — `tojson` сам оборачивает строку в `"`,
что обрывало атрибут раньше времени и ломало клик для **любого** блюда
(не зависело от конкретного названия). Правило на будущее: `|tojson`
безопасен только внутри `<script>` (присваивание JS-переменной), не
внутри инлайновых HTML-атрибутов вида `onclick="..."` — там нужно
`data-*` атрибуты (штатное Jinja-экранирование `{{ }}`) +
`addEventListener` в JS, как уже сделано у автокомплита в этом же файле
и в `writeoff_meal_form.html`. Проверено вживую в браузере (Playwright,
16.07) — клик по чипу добавляет блюдо, ошибок в консоли нет. Задеплоено.

## Режим просмотра/редактирования по дням (согласовано 15.07, реализовано 16.07)

`/menu/` переведён на тот же паттерн, что уже есть у Расходов/карточки
ребёнка/Посещаемости (просмотр→«Изменить»→правка): день с уже сохранённым
меню (`day.has_entries` в `app/routers/menu.py`) открывается read-only
(список блюд текстом по приёмам пищи) с кнопкой «✎ Редактировать»; пустой
день — сразу в редактируемом виде без доп. клика. Переключение через
`toggleDay(dayId, editing)` в `menu/form.html` — по одному состоянию на
каждый день (не общий toggleBlock, как на других страницах, т.к. на
странице сразу несколько дней). Общая кнопка «Сохранить меню» одна на
весь диапазон — скрытые поля чипов остаются в DOM независимо от
view/edit состояния дня, так что просмотровые дни не теряют данные при
сохранении. Проверено вживую (Playwright): переключение в edit сохраняет
уже выбранное блюдо.

## Три реальных бага (23.07) — найдены по живым отчётам Абдусаттара, все задеплоены

**1. Fuzzy-match `get_or_create_dish()` ложно сливал блюда — защита от опечаток
работала только в одну сторону.** Порог `FUZZY_AUTO_MATCH=85` на `WRatio`
переключается на partial-ratio при сильном расхождении длин **в любую
сторону** — не только «длинный текст против короткого блюда», но и наоборот.
Реальный кейс: «Рыбьи биточки на пару» (21 симв.) против уже существующего
«. Овсяная каша на молоке со сливочным маслом» (44 симв., с лишней точкой —
видимо, старый артефакт OCR/опечатки) дало score 85.5 — блюдо Махабат
молча подменялось овсянкой. Первый фикс сессии проверял длину только
одним направлением (кандидат не короче 60% ввода) и пропустил этот
зеркальный случай. Финальная защита — симметричная: `min(len_a, len_b) >=
0.6 * max(len_a, len_b)`. Оба направления проверены на реальных строках
(включая regression на настоящих опечатках — "Каша ристовая"→"каша
рисовая", "Булочька"→"Булочка" — короткие расхождения по-прежнему
матчатся, не задеты).

**Урок:** при защите от одного направления bias'а fuzzy-matching —
всегда проверять симметричность вручную на реальных строках, не
полагаться на интуицию «длинное не должно матчиться на короткое» (в обе
стороны работает одинаково).

**2. Гонка автокомплита.** Debounce-запрос `/menu/dishes/search` (150мс)
не отменялся при нажатии Enter — если ответ приходил уже после того, как
поле очищено и чип добавлен, `render()` всё равно показывал дропдаун с
устаревшей подсказкой на уже добавленное блюдо. Случайный клик рядом
(следующий ввод) добавлял дубликат. Фикс: `render(query, items)` сверяет
`query` с текущим `input.value.trim()` перед показом — ответ на неактуальный
запрос молча игнорируется. Плюс `clearTimeout(timer)` при Enter/выборе
подсказки, чтобы не тратить лишний запрос.

**3. Гонка сохранения дня — настоящая причина большинства дублей на
проде.** Autosave шлёт `POST /menu/day` на каждое добавление блюда, без
очереди. Если Махабат быстро правит разные приёмы пищи одного дня подряд,
уходит несколько overlapping запросов на одну дату; `menu_day_save()`
делает «удалить всё за день → вставить заново» без какой-либо
сериализации — оба запроса видят «нечего удалять» и **оба** вставляют
свою копию, задваивая весь день (не одно блюдо — весь список каждого
приёма пищи целиком). Подтверждено детерминированно: 5 параллельных
запросов без защиты → 5 копий каждой строки; с фиксом — ровно 1 копия
(тест с искусственной задержкой между delete и insert, чтобы гарантированно
поймать окно гонки, а не полагаться на удачное совпадение по времени).

Фикс — `pg_advisory_xact_lock(org_id, date_as_int)` в начале
`menu_day_save()`, до `DELETE`. Держится до конца транзакции (до
`db.commit()`), снимается сам. Разные дни/объекты друг друга не блокируют
(разные ключи лока).

**Диагностика бага №1 и №3 велась через реальные прод-данные**, не
гипотетически: `/menu/dishes/search` — уже публичный HTTP-эндпоинт,
позволяющий проверять fuzzy-совпадения на настоящем каталоге блюд без
прямого доступа к БД (доступ к прямому Postgres прод-соединению из этой
сессии заблокирован классификатором auto-режима — см.
[[architecture/deploy_ops]]). Полная выгрузка всех недель меню (через
залогиненную сессию + regex-парсинг HTML) вскрыла реальный паттерн
дублей — **целые списки приёмов пищи повторялись** (не единичные
блюда), что и указало на гонку сохранения, а не на автокомплит.

**Данные почищены одноразовым скриптом** (20/23/27.07) через тот же
`/menu/day` эндпоинт, что использует форма — не прямой SQL. «Рыбьи
биточки на пару» не восстановлены: блюдо никогда не сохранялось под
своим именем, Махабат ввела его заново после фикса.

## Проверено (10.07, Playwright, локально)

- Создание меню на неделю, чипы, автокомплит, сохранение — 0 ошибок
  консоли, баннер-предупреждение корректно сдвигается на первый
  незаполненный день
- Повторное открытие страницы после сохранения — блюдо появляется и как
  сохранённый чип, и в «обычно даём» для этого приёма пищи
- Списание на приём пищи с привязкой к блюду — `WriteOff.dish_id`
  сохраняется корректно; без меню на дату — форма не блокируется
- `pytest` — без новых регрессий (те же 2 доеденных падения, не связанные
  с этим модулем)

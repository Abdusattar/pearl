# Деплой и эксплуатация — Railway

> Обновлено: 2026-07-05

## Пайплайн деплоя

`git push origin master` → Railway собирает `Dockerfile` → контейнер стартует
через `entrypoint.sh`:

```sh
alembic upgrade head
uvicorn app.main:app --host 0.0.0.0 --port ${PORT:-8000}
```

Миграции применяются **автоматически при каждом деплое**, без ручного шага.
Значит: закоммитить + запушить = задеплоено и смигрировано. Если что-то
закоммичено локально, но не запушено — на проде его физически нет, даже если
`git log` локально показывает коммит как «сделанный».

**Разрыв 04.07→05.07:** сессия 04.07 закоммитила работу (карточка ребёнка,
ФИО по частям), но не запушила. Это пролежало незамеченным сутки — `git log
origin/master..master` было бы способом поймать это сразу. Стоит проверять
перед закрытием сессии, если были коммиты.

## Диагностика прод-окружения — Railway CLI

Установлен и залинкован локально (`railway status` внутри `D:\aispace\pearl`
работает без доп. настройки). Полезные команды:

- `railway status --json` — какой коммит реально задеплоен (`commitHash`),
  какой план (`plan`), статус сервиса
- `railway metrics` — CPU/RAM/сеть за последний час + разбивка HTTP-кодов
  (2xx/3xx/4xx/5xx) + p50/p90/p95/p99 latency + список последних деплоев
- `railway logs` — логи контейнера (stdout/stderr), включая полные traceback
  необработанных исключений

Это быстрее и точнее, чем гадать по симптомам («медленно») — метрики сразу
показывают, ресурсы это или ошибки.

## Инцидент 05.07: «медленный прод» оказался багом, не инфраструктурой

Жалоба «не нравится скорость на проде» разобрана по шагам:

1. Проверили тариф — `plan: "trial"` (shared vCPU, 1GB RAM). Похоже на
   очевидную причину, но:
2. `railway metrics` показал CPU ~0%, память 10% от лимита — ресурсов с
   большим запасом. **Не бутылочное горлышко.**
3. Latency p50–p99 = 4мс — сервер отвечает мгновенно. **Не сетевая/инфра
   медленность на уровне приложения.**
4. Зато **31% запросов — 5xx** за последний час. Это не «медленно», это
   «падает». `railway logs` показал точный traceback:
   `AttributeError: 'NoneType' object has no attribute 'role'` в
   `get_accessible_orgs()`.

**Корень:** `app/routers/expenses.py` — 9 из 10 обработчиков вызывали
`get_current_user(request, db)` и сразу использовали результат, не проверяя
на `None`. Все остальные роутеры (`students.py`, `income.py`, `reports.py`)
и 1 обработчик в самом `expenses.py` — используют правильный паттерн:

```python
user = get_current_user(request, db)
if not user:
    return RedirectResponse("/login", status_code=302)
```

При истёкшей сессии (или прямом заходе по ссылке без логина) это давало
`500` вместо тихого редиректа на логин. Пользователь воспринимал это как
«тормозит» — реально страница просто падала и приходилось перезаходить.

**Урок:** при жалобе на «медленно» — сначала `railway metrics` (ресурсы?
латентность? код ошибок?), а не сразу разговор о смене инфраструктуры/региона.
Смена облака/тарифа ничего не чинит, если причина — необработанный `None` в
коде.

**Конвенция на будущее:** каждый новый обработчик, вызывающий
`get_current_user()` напрямую (а не через готовый dependency), обязан сразу
проверить на `None` и редиректнуть на `/login`. Это легко забыть при
копировании существующего кода в `expenses.py` — именно так появился баг.

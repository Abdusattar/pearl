# Деплой и эксплуатация — Railway

> Обновлено: 2026-07-05

## Пайплайн деплоя

`git push origin master` → Railway собирает `Dockerfile` → контейнер стартует
через `entrypoint.sh`:

```sh
alembic upgrade head
uvicorn app.main:app --host 0.0.0.0 --port ${PORT:-8000}
```

Миграции применяются **автоматически при каждом деплое**, без ручного шага.
Значит: закоммитить + запушить = задеплоено и смигрировано. Если что-то
закоммичено локально, но не запушено — на проде его физически нет, даже если
`git log` локально показывает коммит как «сделанный».

**Разрыв 04.07→05.07:** сессия 04.07 закоммитила работу (карточка ребёнка,
ФИО по частям), но не запушила. Это пролежало незамеченным сутки — `git log
origin/master..master` было бы способом поймать это сразу. Стоит проверять
перед закрытием сессии, если были коммиты.

## Диагностика прод-окружения — Railway CLI

Установлен и залинкован локально (`railway status` внутри `D:\aispace\pearl`
работает без доп. настройки). Полезные команды:

- `railway status --json` — какой коммит реально задеплоен (`commitHash`),
  какой план (`plan`), статус сервиса
- `railway metrics` — CPU/RAM/сеть за последний час + разбивка HTTP-кодов
  (2xx/3xx/4xx/5xx) + p50/p90/p95/p99 latency + список последних деплоев
- `railway logs` — логи контейнера (stdout/stderr), включая полные traceback
  необработанных исключений

Это быстрее и точнее, чем гадать по симптомам («медленно») — метрики сразу
показывают, ресурсы это или ошибки.

## Инцидент 05.07: «медленный прод» оказался багом, не инфраструктурой

Жалоба «не нравится скорость на проде» разобрана по шагам:

1. Проверили тариф — `plan: "trial"` (shared vCPU, 1GB RAM). Похоже на
   очевидную причину, но:
2. `railway metrics` показал CPU ~0%, память 10% от лимита — ресурсов с
   большим запасом. **Не бутылочное горлышко.**
3. Latency p50–p99 = 4мс — сервер отвечает мгновенно. **Не сетевая/инфра
   медленность на уровне приложения.**
4. Зато **31% запросов — 5xx** за последний час. Это не «медленно», это
   «падает». `railway logs` показал точный traceback:
   `AttributeError: 'NoneType' object has no attribute 'role'` в
   `get_accessible_orgs()`.

**Корень:** `app/routers/expenses.py` — 9 из 10 обработчиков вызывали
`get_current_user(request, db)` и сразу использовали результат, не проверяя
на `None`. Все остальные роутеры (`students.py`, `income.py`, `reports.py`)
и 1 обработчик в самом `expenses.py` — используют правильный паттерн:

```python
user = get_current_user(request, db)
if not user:
    return RedirectResponse("/login", status_code=302)
```

При истёкшей сессии (или прямом заходе по ссылке без логина) это давало
`500` вместо тихого редиректа на логин. Пользователь воспринимал это как
«тормозит» — реально страница просто падала и приходилось перезаходить.

**Урок:** при жалобе на «медленно» — сначала `railway metrics` (ресурсы?
латентность? код ошибок?), а не сразу разговор о смене инфраструктуры/региона.
Смена облака/тарифа ничего не чинит, если причина — необработанный `None` в
коде.

**Конвенция на будущее:** каждый новый обработчик, вызывающий
`get_current_user()` напрямую (а не через готовый dependency), обязан сразу
проверить на `None` и редиректнуть на `/login`. Это легко забыть при
копировании существующего кода в `expenses.py` — именно так появился баг.

## Разовые скрипты против прод-БД (например, добавление пользователя)

`railway run python scripts/xxx.py` на этой Windows-машине падает с
`UnicodeDecodeError` при разборе DSN — похоже на баг самого Railway CLI при
инъекции переменных окружения в дочерний процесс на Windows. Обходной путь —
не оборачивать в `railway run`, а взять адрес БД напрямую и передать через
`DATABASE_URL` явным env var перед `python`:

```
railway variables --service Postgres --kv   # смотреть DATABASE_PUBLIC_URL
DATABASE_URL="postgresql://postgres:<pass>@<host>.proxy.rlwy.net:<port>/railway" \
  PYTHONIOENCODING=utf-8 python scripts/xxx.py
```

Важно: брать **`DATABASE_PUBLIC_URL`**, не `DATABASE_URL` из сервиса Postgres —
последний указывает на `postgres.railway.internal`, недоступный за пределами
Railway-сети. `PYTHONIOENCODING=utf-8` обязателен — иначе `print()` с кириллицей
роняет скрипт `UnicodeEncodeError` на консоли с кодировкой cp1251 (тот же класс
проблемы, что и с bash/curl — см. `context/summary.md` → «Готча (кириллица)»).

**Sequence-баг ловится и на проде, не только в dev:** `users_id_seq` (и,
вероятно, `organizations_id_seq`) был рассинхронизирован ещё в dev-сессии 04.07
(seed с явными id без `setval`) — 06.07 та же проблема всплыла на **проде** при
попытке добавить нового пользователя (`UniqueViolation: Key (id)=(1) already
exists`). Чинится одноразово:

```sql
SELECT setval('users_id_seq', (SELECT MAX(id) FROM users));
```

Если где-то ещё сидировали записи с явным `id` вручную — тот же класс бага
может всплыть в других таблицах.

## Весь `alembic upgrade head` — одна транзакция, не одна на ревизию (13.07)

`alembic/env.py` оборачивает `context.run_migrations()` в один
`with context.begin_transaction():` **вокруг всего вызова** `upgrade head`,
а не по одной транзакции на каждую ревизию. Значит если из 12 накопленных
непримененных миграций сломается последняя — откатятся **все 12**, БД
останется ровно в том состоянии, что была до деплоя. Подтверждено на
практике 13.07: пуш 13 коммитов/12 миграций упал на последней (products
FK), контейнер не поднялся (502 несколько минут), но `alembic_version` на
проде остался на старом значении — ни одна из 12 миграций не закоммитилась
частично. Фикс + повторный пуш прошли чисто.

**Практическое следствие:** большой пакет миграций либо применяется
целиком, либо не применяется вообще — не нужно бояться "частичного"
состояния БД после упавшего деплоя. Но контейнер всё равно не поднимется,
пока миграция не пройдёт — значит прод недоступен (502) на всё время между
упавшим деплоем и следующим успешным пушем. Чем больше пакет непроверенных
вместе миграций — тем выше риск такого простоя. Стоит по возможности не
копить миграции на несколько сессий, деплоить чаще меньшими пакетами —
но 13.07 сознательно копили (см. `context/summary.md`), тогда компенсировать
это должно **тщательной проверкой каждой миграции на реальных данных
прода перед пушем** (не только на dev), особенно там, где миграция
удаляет строки.

**Конкретная ошибка 13.07:** миграция, удаляющая старые категории расходов
(`o5p6q7r8s9t0`), проверяла на существование ссылок только `transactions`,
забыв, что `products.expense_category_id` — тоже FK на `expense_categories`.
На dev таких товаров не было (случайно), на проде — 2 (заведены до
появления подкатегорий). **Урок:** перед миграцией, удаляющей строки из
таблицы-справочника, искать **все** FK на неё в `app/models.py`
(`grep 'ForeignKey("expense_categories'` или аналогично для любой другой
таблицы), не только те, что кажутся очевидными по смыслу задачи.

# Модуль биллинга (долги/переплаты)

> Зафиксировано: 2026-07-03

## Ключевое ограничение, определившее архитектуру

Протокол банка (Optima, OSMP/QIWI) не передаёт назначение платежа — родитель
вводит PIN и платит сумму, без выбора «за что» (учёба/проезд/секция). Значит
система в принципе не может узнать, за какую конкретно услугу пришли деньги.

**Решение: один общий баланс на ребёнка**, а не раздельный учёт по видам услуг.
Альтернативой было бы завести отдельный PIN на каждую услугу — отклонено, это
усложнило бы жизнь родителю (несколько кодов) ради выгоды, которая не нужна
школе на практике.

## Модель данных

- **Учёба** — **переделано 13.07** (было `Student.extra.monthly_fee`,
  индивидуальное число на каждого ребёнка — см. раздел ниже про то, почему
  это оказалось неверной моделью). Теперь: `Service` с флагом `is_tuition=True`
  (одна такая услуга на объект, цена = базовый тариф, редактируется на
  `/services/` как у любой услуги, применяется автоматически всем активным
  детям объекта — не через чекбокс) + `Student.discount_amount` (скидка в
  сомах персонально ребёнку, не процент). Эффективный тариф = цена
  `is_tuition`-услуги минус `discount_amount`, посчитано в
  `services/billing.py::_tuition_fee()`.
- **`Service`** — справочник услуг с ценой, привязан к `organization_id`
  (цена может отличаться школа/садик). Два вида: `is_tuition=True` (ровно
  одна на объект, обязательная для всех, без чекбокса, удаление
  заблокировано на сервере) и обычные опциональные (проезд, секции —
  подключаются по ребёнку через `StudentService`)
- **`StudentService`** — подключение услуги ребёнку, с историей (`start_date`/
  `end_date`, тот же паттерн что `Enrollment` для групп) — включил/выключил
  чекбоксом на карточке ребёнка
- **`Charge`** — начисление: `student_id`, `amount`, `description`, `date`.
  Единственный источник — автоматическое начисление раз в месяц, сумма = тариф
  + все активные `StudentService`. **Ручной корректировки (произвольная
  сумма+описание вручную) в системе нет — убрана целиком 04.07**, см. ниже.
- **Баланс** = `сумма(Charge)` − `сумма(Transaction where type=income)` по
  `student_id`. Положительное = долг, отрицательное = переплата

## Генерация начислений — без cron

`generate_monthly_charges()` в `app/services/billing.py` вызывается при каждом
заходе на `/students/` или `/reports/` — идемпотентно (проверяет, есть ли уже
начисление за этот месяц по `description == "Начисление за месяц"`). Тот же
паттерн, что и авто-архивация выбывших студентов (`archive_stale_students`) —
никакого планировщика/cron не требуется для простых периодических задач в
этом проекте, достаточно проверки на чтении.

## Структура интерфейса (IA) — обновлено 04.07

Изначально баланс и тариф были прямо в списке `/students/` — пользователь
справедливо указал, что это превращает список в свалку цифр. Дальше принцип
«у каждого вида данных — одно чёткое место» довели до уровня отдельных
страниц/форм, не только списка:

| Место | Что показывает |
|---|---|
| `/students/` | Только личные данные: PIN, ФИО, группа, статус (+фильтр по группе/статусу). Вся строка кликабельна → карточка |
| `/students/{id}/edit` (карточка) | Два независимых блока с раздельным сохранением: «Личные данные» (ФИО по частям, группа, статус, контакты родителя) и «Оплата» (тариф, доп.услуги). Денег и истории здесь больше нет |
| `/students/{id}/history` | Баланс + единая хронологическая лента начислений и оплат (цвет/знак различает начисление vs оплату, бейдж Optima/Вручную у оплат) |
| `/reports/` | Сводный долг/переплата по ВСЕМ детям — фильтр по группе + чекбокс «Только с долгом», итоговые суммы сверху |
| `/income/` (Оплаты) | Список платежей с колонкой текущего баланса ребёнка; клик по строке → `/history` |

**Важный паттерн, введённый 04.07 — просмотр/редактирование как отдельные
режимы.** И «Личные данные», и «Оплата» на карточке по умолчанию открываются
как текст (не поля ввода) — кнопка «Изменить» переводит в редактируемый режим,
«Сохранить» спрашивает подтверждение, «Отмена» откатывает без запроса на
сервер. Причина: Айжан и Мунара нетехнические, боятся случайно испортить
данные — редактирование должно быть осознанным действием, не побочным
эффектом открытия страницы. Использовать этот же паттерн для будущих форм
редактирования данных, которые правятся редко, но просматриваются часто.

## Ручная корректировка — убрана целиком (04.07)

Была форма «добавить начисление» с произвольной суммой (+ долг / − списание)
и текстовым описанием — для старых долгов и разовых скидок. Убрана по
инициативе владельца: эндпоинт был доступен любому залогиненному пользователю
без ограничения роли, а у `Charge` не было `created_by` — то есть кто угодно
мог списать долг ребёнку без единого следа, кто это сделал. Решение — не
чинить (роль+аудит), а убрать функцию целиком. Если понадобится занести
старый долг/скидку — сейчас в системе для этого штатного механизма нет.

## Статус ребёнка "Заморожен" — реализовано 16.07

Родитель договаривается на паузу (месяц-два), место в группе остаётся за
ребёнком — не отдаётся другому. Экономика (объяснено Абдусаттаром 16.07):
если место не резервировать, его займёт другой ребёнок и школа заработает
на нём полную сумму — значит 50% (не 0%, не 100%) это компромисс за
удержание места. "Бронь" и "Заморожен" — один и тот же механизм под
разными словами (подтверждено 16.07), не два статуса.

Пока актуально **только для Садика Сокулук** (подтверждено Махабат
16.07) — Школа и Кожомкул, возможно, вообще не будут им пользоваться,
возможно позже понадобится что-то своё. `Student.status` — обычная
строка без CHECK-constraint в БД, так что добавить ещё один статус
для них позже, если появится конкретный запрос — не переделка, а просто
новое значение.

**Модель:** `Organization.frozen_discount_percent` (Numeric, default 50)
— настройка объекта, не константа в коде: софт планируется другим
садикам/школам, у каждого объекта могут быть свои цифры (или этот статус
вообще не использоваться). Редактируется на `/services/` теми же, кто
меняет тариф (`PRICE_EDITORS` — Айдай/Талас), через
`POST /services/frozen-percent`.

**Начисление** (`billing.generate_monthly_charges`): при `status=='frozen'`
— `tuition_fee * frozen_discount_percent / 100`, без доп.услуг (ребёнок
ими не пользуется), без пропорции по дате (упрощение — заморозка средины
месяца не пересчитывается по дням).

**Enrollment НЕ закрывается** при заморозке (`update_student` — ветка
`status == "frozen": pass`, ничего не трогает) — место в группе остаётся
занятым за ребёнком, в отличие от `inactive`, где Enrollment закрывается.

**Optima** (`optima.py`): `status not in ("active", "frozen")` → отказ —
заморожённый ребёнок должен иметь возможность оплатить свои 50% через банк
тем же PIN.

**Отчёт должников** (`reports.py`) включает `frozen` наравне с `active` —
у заморожённого ребёнка тоже может быть долг.

**Месяц самой заморозки не пересчитывается** (идемпотентность
`generate_monthly_charges` по месяцу) — скидка начинает действовать
только со следующего начисления. Осознанно принято 16.07: возвращать
ручную корректировку начислений (убрана 04.07 из-за отсутствия аудита)
ради разовой переплаты за один месяц — дороже, чем сама проблема.
Решение с родителем в такой месяц — устно, вне системы.

## Скидка за дни отсутствия — решено 10.07, не реализовано

Вопрос висел с 08.07 (см. [[architecture/warehouse_module]], гап-анализ
09.07 — «Корзина B»). Ответ Мунары: **перерасчёт минус 100 сом за каждый
день пропуска**, фиксированная сумма, не пропорционально тарифу.

Источник дней пропуска — уже готовый `Attendance` (09.07, `present=False`).
Реализация ещё не спроектирована — открытые вопросы для отдельного
блюпринта: считать ли по всем детям одинаково 100 сом, или учитывать, что у
части детей другой тариф; входит ли перерасчёт в `Charge` при генерации
(значит `generate_monthly_charges()` должен читать `Attendance` за период),
или это отдельная сущность-корректировка (но ручная корректировка была
убрана целиком 04.07 из-за отсутствия аудита — если возвращать что-то
похожее, обязательно с `created_by` и не давать произвольную сумму, только
детерминированный расчёт от `Attendance`).

## Тариф — реальная услуга + скидка в сомах (13.07, была свободным числом)

**Как обнаружилось:** `Student.extra.monthly_fee` изначально задумывался как
«индивидуальный тариф на ребёнка», но по факту у 96 детей Сокулука было
всего 5 значений (8000×40, 7500×28, 5000×6, 5500×2, 4000×8) + 12 детей
вообще без тарифа. Абдусаттар подтвердил реальную механику: 8000 — базовый
тариф по умолчанию для всех, отклонения — это **скидка в сомах**
конкретному ребёнку (500/3000/2500/4000 соответственно), не другой тариф и
не проценты.

**Почему не подошёл уже существующий `discount_percent`:** заведён 12.07
специально под скидку на тариф, но (а) процентный, а не в сомах, (б) форма
карточки ребёнка ни разу не получила полей ввода для него — заготовка на
сервере была, ввести скидку через UI было физически нельзя. Отсюда 0 у всех
96 детей на момент обнаружения — не «скидок нет», а «поле было
недоступно».

**Почему тариф стал `Service`, не константой в коде:** тариф будет расти
(подтверждено), и менять его нужно обычной правкой цены на `/services/`, не
деплоем кода. `is_tuition=True` — единственный маркер, по объекту, а не по
имени строки (в отличие от `SERVICE_CAT_NAMES`/`RECURRING_ONLY_NAMES` в
`expenses.py`, оба — известный техдолг, здесь сознательно сделано через
явный флаг).

**Миграция существующих данных** (`p6q7r8s9t0u1`, `q7r8s9t0u1v2`):
`discount_amount = 8000 - monthly_fee` для тех, у кого тариф был, `0` для
12 без тарифа (раньше они не получали начисление вообще —
`_tuition_fee() == 0` → `total <= 0` → `generate_monthly_charges()`
пропускал их молча). Проверено на реальных 96 детях после пересчёта —
распределение начислений совпало с исходным один в один, никого не задело
задним числом.

## Известный риск — тарифы не сверены с Айжан/Мунарой напрямую

Механика подтверждена **Абдусаттаром** (13.07), но не самими Мунарой/Айжан
— это разговор, который тянется с 03.07 и до сих пор не случился. Прежде
чем показывать суммы долга кому-то как окончательный факт — свериться. Раз
теперь известна точная механика (база 8000 + скидка в сомах), сверку можно
сделать конкретно: «правда ли база 8000, и правильные ли суммы скидок у
конкретных детей» — не абстрактный вопрос «какие тарифы», как раньше.

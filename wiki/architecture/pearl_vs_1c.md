# Архитектура: Pearl IS vs 1С — Разграничение ответственности

> Зафиксировано: 18.06.2026
> Источник: диалог с финэкспертом (NotebookLM CFO) + решение с Абдусаттаром

## Операционная модель

Pearl IS и 1С:Бухгалтерия КР работают параллельно. Это НЕ дублирование — у каждой системы своя зона ответственности.

### Pearl IS = Источник правды для операционных данных

| Данные | Как попадают | Кто работает |
|---|---|---|
| Доходы (оплаты) | Optima webhook (автоматически) | — |
| Расходы | OCR чеков | Айжан, Мунара |
| Физический склад (кг, порции) | Форма прихода/списания | Повар, завхоз, Мунара |
| Дети, посещаемость | Ввод / импорт Excel | Воспитатели |

### 1С = Расчётная и отчётная система

| Данные | Источник |
|---|---|
| Начисление зарплаты | Бухгалтер (расчёт в 1С) |
| НДФЛ, Соцфонд | Автоматически из зарплаты |
| Годовая отчётность | Из данных 1С |
| Бухгалтерские проводки | Из сводки Pearl |

## Ежемесячная передача Pearl → бухгалтер

До 1-го числа следующего месяца Pearl передаёт 4 документа:

| Документ | Содержание | Ключевая аналитика |
|---|---|---|
| **Реестр реализаций** | ФИО ребёнка, вид услуги, локация, сумма за месяц | Субконто: номенклатура + подразделение |
| **Авансовый отчёт** | Чеки с категорией расхода, поставщик, ИНН | Субконто: статья затрат + подразделение |
| **Акты ТМЗ** | Закупка и списание продуктов (кг, цена) | Количественный учёт |
| **Ведомость взаиморасчётов** | Остаток аванса по каждому ребёнку (начало → оплата → зачёт → остаток) | Субконто: контрагент + договор |

Источник требований: NotebookLM CFO эксперт (НК КР + НСФО КР + Положение по бухучёту).

## Принцип "следуй за оператором"

Правило выбора системы: **данные принадлежат там, где работает человек, который их создаёт**.

- Расходы → Pearl (Мунара/Айжан фотографируют чеки)
- Склад → Pearl (повар принимает товар, не бухгалтер)
- Зарплата → 1С (бухгалтер считает gross→net с КР ставками)

## Ключевые учётные правила

### Авансы от родителей
```
При получении оплаты:   Дт Банк → Кт Авансы полученные (ОБЯЗАТЕЛЬСТВО)
Ежемесячно (признание): Дт Авансы → Кт Выручка (по факту оказания услуги)
```
Ошибка: признавать годовую оплату выручкой в момент поступления → искажает P&L.

### Складской учёт (физический)
- Pearl хранит: приход (кг, цена), списание (кг, кол-во детей)
- Остаток = SUM(приходы) − SUM(списания) per product per org
- children_count на списании → себестоимость питания на 1 ребёнка в день
- 1С получает итоговые акты (суммы), не каждую строку

### Выручка по видам
Необходим раздельный учёт: **обучение** vs **питание** — разные налоговые режимы по НК КР.

## Что должна хранить Pearl (минимум для передачи бухгалтеру)

**Transaction:**
- `service_type` — обучение / питание (добавить)
- `supplier_inn` — ИНН поставщика (добавить для расходов)
- `receipt_number` — номер чека ККМ

**WarehouseReceipt (уже есть):**
- product, qty, price_per_unit, total_cost, organization_id, supplier_name, date

**WriteOff (уже есть):**
- product, qty, organization_id, children_count, date

**Логика признания выручки (не реализована):**
- Таблица `revenue_recognitions` или расчёт из авансов — следующий спринт

## Хранение документов (требование КР)
Все первичные документы хранятся 6 лет. Pearl хранит цифровые копии (фото чеков). Оригиналы физически передаются бухгалтеру для архивации.

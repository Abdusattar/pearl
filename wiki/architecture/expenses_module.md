# Модуль Расходы — архитектура

## Поток данных

```
Фото → upload → Receipt (file_path, file_hash) → OCR → ReceiptItems
                                                        ↓
                                             confirm → Transaction
                                                        ↓
                                               ReceiptTransaction (связка)
```

## Ключевые решения

**file_hash (SHA256)** — защита от дублей. Если то же фото загружается повторно (бот + веб), находим существующий Receipt и переиспользуем.

**Фото на диске, путь в БД** — `receipts.file_path` = relative path. FastAPI отдаёт `media/` как статику. При переносе в облако: заменить запись файла на S3 upload в одном месте (`handle_upload`).

**Manual receipts** — ручные записи без фото создают Receipt с `file_path='manual'`, `ocr_status='manual'`. Это позволяет хранить позиции (ReceiptItems) без реального файла, не меняя схему.

## Справочник продуктов — архитектура (решено 27.06.2026)

### Два уровня продуктов

```
Product.is_standard = True   → эталонный (39 реальных продуктов из OCR-каталога)
Product.is_standard = False  → временный (создаётся автоматом из незнакомых OCR-строк)
```

**Почему `is_standard` флаг, а не `category IS NULL`:** авто-промоут не знает product.category в момент первого появления. Явный bool надёжнее.

**Эталонный каталог:** `data/standard_products.json` — 101 продукт (обновлено 06.07, было 39), реально встреченных в 109 квитанциях. Загружается через `scripts/load_standard_catalog.py`. Категории: мясо, овощи, зелень, крупы, молочные, специи, масла, фрукты, хлеб, напитки — плюс с 06.07 **стройматериалы** (30), **бытовая химия** (10), **игрушки** (5), **инвентарь** (2, батарейки/вилки). До 06.07 каталог был только про еду — не-食 категории (Ремонт/Хозяйство) шли только агрегированной суммой, без разбора по позициям.

**Принцип наименования (06.07):** эталон = вид товара, не марка/сорт/бренд — «Цемент», не «Цемент М400»; «Саморез» объединяет и «Шуруп» (для нашего учёта разница не важна). Но разводим по смыслу, если это реально разные товары: «Болгарский перец» ≠ «Перец чёрный» (овощ vs специя), «Кефир» ≠ «Молоко», «Клей» ≠ «Клеевой пистолет» (материал vs инструмент — ошибка первой итерации 06.07, слили в одни алиасы, потом развели).

**Каталогизировать сразу все позиции, не ждать порога ≥2 упоминаний** — если чеки уже отсканированы (лежат в `data/ocr_scan_cache.json`), нет смысла ждать второго появления редкого товара, можно сразу решить стандартное имя. Единственное, что не каталогизировать — нечитаемый OCR-мусор (обрывки слов с плохого почерка) и разовые нетипичные покупки (мебель и т.п. — это не расходник, а актив).

**Открытые позиции без решения (06.07):** «Лесна» — назначение не разгадано; «Кабель-канал» — слит в алиасы «Гофра», возможно стоит развести отдельно; «Пепси»/«Кола» — заведён генерик «Напиток газированный» на будущее, если появятся регулярно.

**Известный gap в скрипте загрузки:** `scripts/load_standard_catalog.py` только ДОБАВЛЯЕТ новые алиасы, никогда не переназначает уже существующие — если алиас уже привязан к продукту A, а в новой версии JSON он должен указывать на продукт B, надо руками чистить `ProductAlias` в БД (случилось 06.07 с «Автомат эл»/«Изолента», ошибочно слитыми с «Кабель» в первой итерации).

**Временный продукт:** создаётся через `get_or_create_product()` с `is_standard=False`. Никогда не промоутируется автоматически для категорий Ремонт / Транспорт / Прочее.

### Авто-промоут — убран (08.07), промоут теперь только вручную

**До 08.07:** товар, встретившийся в чеках 3 раза, сам становился эталоном
(`maybe_promote()` в `services/products.py`, вызывался из `handle_confirm`).

**Причина отмены:** Абдусаттар решил, что промоут во всех случаях должен идти
только через его собственное решение (в диалоге с Claude по мере накопления
временных товаров), а не автоматически по счётчику повторов — риск, что
случайная опечатка или неудобный вариант написания, повторившись трижды,
зацементируется в каталоге как «эталон», выше, чем польза от автоматизации.
Функция `maybe_promote()` и её вызов удалены из кода целиком (не оставлена
мёртвым кодом) — если понадобится вернуть, история в git.

**Как теперь становится эталоном:** только руками — обновлением
`Product.is_standard` (тем же путём, каким собирался каталог 39→101 позиций
06.07: смотрим на накопленные временные товары, решаем какие из них —
реальные повторяющиеся позиции, а какие — мусор/дубли/опечатки). Периодическая
задача, не привязана к конкретной странице в UI — Абдусаттар просит Claude
свести список временных товаров и обсудить, когда накопится достаточно.

### Дедупликация OCR-опечаток

Gemini-матчер видит ОБА типа (стандартные и временные). "корианд" и "кореанд" → оба матчатся к одному временному продукту → счётчик растёт корректно. При промоуте alias сохраняется.

### Структура данных

```
products (name, category, unit, is_standard)
product_aliases (raw_text → product_id)
receipt_items.product_id → products
```

Старые ReceiptItems при промоуте не трогаем — они уже ссылаются на product_id. Когда продукт промоутируется (`is_standard=True`), старые записи автоматически выигрывают.

## Позиции обязательны везде, кроме услуг (решено 06.07, вторая половина сессии)

Заказчик (Айжан/Мунара) явно потребовали: разбивка по позициям нужна для **всех** товарных
категорий, включая Ремонт (стройматериалы) — иначе потом не посчитать юнит-экономику
(что и сколько реально расходуется). Единственное исключение — категории-услуги
(Транспорт, Прочее), которые по своей природе не раскладываются на "товарные позиции"
(заправка, такси и т.п.) — их решили рассмотреть отдельно позже, не блокируя остальное.

Практическое следствие: `add.html` (ручная запись без фото) показывает блок позиций для
всех категорий кроме Транспорта/Прочего — раньше показывался только для Питания.

Категории-исключения для **авто-промоута** временного товара в эталонный:
`{"транспорт", "прочее"}` (`_is_promote_eligible()` в expenses.py). **Ремонт убран из
этого списка 06.07** — по стройматериалам нужен такой же растущий эталонный каталог,
как по еде (заказчик прямо просил "гвозди" и подобное как стандартные позиции).

## Категория расхода — теперь всегда авто, без ручного выбора (06.07, крупная переделка)

**До:** человек выбирал `ExpenseCategory` вручную в форме (один select, либо ручная
"разбивка по категориям" с несколькими select+amount).

**После:** поле "Категория" убрано из confirm.html и add.html (для фото-квитанций) целиком.
Категория выводится из товара:

```
products.expense_category_id → FK на expense_categories.id   (миграция q5r6s7t8u9v0)
```

Это **отдельное поле от старого `Product.category`** (строка "мясо"/"стройматериалы"/...,
используется для группировки на складе и как AI-подсказка при сопоставлении) — два разных
классификатора для разных целей: `category` — мелкий тип товара, `expense_category_id` —
крупная бухгалтерская статья для отчётности. Бэкфилл при миграции: мэппинг мелкая→крупная
(стройматериалы→Ремонт, бытовая химия→Бытовая химия, игрушки→Игрушки, инвентарь→Хозяйство,
всё остальное food→Продукты питания), проверен на реальных данных перед деплоем (пробный
прогон в отдельной колонке с откатом).

**Авторазбивка при `POST /confirm`:** позиции чека группируются по `expense_category_id`
их товара; на каждую получившуюся группу — своя `Transaction`. Сумма/оплата/долг
распределяются **пропорционально** между группами (последней группе — остаток, без ошибок
округления). Проверено на реальном чеке #4 (смешанный: Бытовая химия + Продукты питания +
позиции без категории, частичная оплата 2000 из 3044) — суммы сошлись копейка в копейку.

**Неопознанная позиция** (новый товар, ещё не в каталоге) → её доля идёт с
`category_id = NULL`, не блокирует проведение. Счётчик "Без категории: N" на `/expenses/`
— чтобы такие случаи не забывались; донастраивается через привязку `expense_category_id`
у товара на `/warehouse/products/` (можно указать сразу при создании нового товара).

**Долг привязан к одному поставщику на чек** (не по категориям) — пользователь подтвердил,
что чек всегда от одного поставщика, так что пропорциональное деление долга между
категориями не создаёт путаницы "кому сколько должны".

## Лента платежей поставщикам — реализовано 16.07

`app/services/supplier_ledger.py` — единственный источник правды про долг поставщику.
Платежи (`SupplierPayment`) НЕ мутируют `Transaction.amount_paid` (то поле остаётся
историческим фактом "сколько оплатили в момент закупа") — баланс считается на лету:
недоплаты по закупам + `Supplier.opening_balance` (начальное сальдо на дату начала учёта)
минус все платежи, FIFO по дате от старых долгов к новым. Из-за этого удаление/правка
платежа не требует логики отката — всё пересчитывается заново на чтении.

Та же функция теперь питает всё, что раньше читало `amount_paid` напрямую: подсветка
долга в `/expenses`, итог "Долг поставщикам", список `/suppliers/` и детальная страница
`/suppliers/{id}` (баланс + история + форма платежа + правка сальдо). Без этого после
первого же платежа старые места продолжали бы показывать погашенный долг как висящий —
именно та несостыковка, которую платежи должны были устранить, а не породить.

**История поставщика группируется по чеку/записи** (`receipt_id` через
`ReceiptTransaction`), не по отдельной `Transaction` — `create_split_transactions` режет
закуп на несколько проводок по категориям расходов, и без группировки один визит к
поставщику выглядел бы как N разных закупов. Группировка — только для отображения ленты;
per-Transaction остаток (для подсветки в `/expenses`) считается отдельно и не тронут.

## "OCR ошибся" — не бывает пропуска позиций (06.07)

Кнопка на confirm.html **не** проводит чек одной суммой без категории — специально, по
прямому требованию заказчика ("нет такого что без позиции закатал"). Она очищает
искажённые OCR-строки (нереальные количества/суммы) и оставляет таблицу позиций открытой
— человек перепечатывает вручную, глядя на бумажный чек. Единственный по-настоящему
беспозиционный путь остался как fallback в бэкенде (`group_totals[None] = amount`, если
позиций вообще не подано) — не поощряется UI, но не ломается, если человек всё же ничего
не впишет.

## AI предлагает стандартизированное имя для новых позиций (06.07)

Раньше для нераспознанной позиции показывался сырой OCR-текст ("Говядина тушёная высший
сорт Казахская 500гр"), и персонал печатал стандартное имя с нуля — риск, что неопытный
сотрудник насорит в каталог (опечатки, случайные варианты). Теперь `normalize_items()`
(`services/normalize.py`) при отсутствии совпадения с каталогом **дополнительно** просит
у Gemini `clean_name` — короткое обобщённое имя в стиле каталога (без бренда/сорта/веса).
Хинт показывает и предложение, и исходный OCR-текст рядом: `«Овощи», в каталоге нет
(OCR: «овещи»)`. Если строка вообще не товар (например, "за дорогу" в рукописной заметке)
— ИИ не выдумывает, `clean_name = null`, обычное "проверь — новая позиция". Финальное
решение (принять/поправить) всегда за человеком — авто-принятие сознательно не сделано,
см. следующий раздел про причину.

**Почему нельзя автопринимать даже уверенную AI-догадку:** цепочка
`AI clean_name → get_or_create_product → maybe_promote (после 3 повторов → эталон)`
означает, что одна тихо принятая ошибка не останется на одном чеке — она станет целью
сопоставления для будущих чеков и через 3 повтора зацементируется в каталоге как эталон,
неотличимо от правильной записи.

## Два состояния на confirm-странице (упрощено 06.07, было 4)

Изначально было 4 цветных состояния (зелёный/жёлтый/оранжевый/красный, по внутреннему
типу AI-совпадения: exact alias / fuzzy-эталон / fuzzy-временный / не найдено). Персонал
(и сам Абдусаттар) путался — эдвайзер указал на причину: 4-уровневая градация кодирует
*внутреннюю* логику системы, а сотруднику нужно бинарное решение «доверять строке или
проверить». Свели к 2:

| Состояние | Рамка | Хинт | Что значит |
|---|---|---|---|
| `product_matched` (точный alias) | Зелёная (2px) | ✓ в каталоге | Ничего делать не надо |
| всё остальное | Жёлтая (2px) | «проверь — похоже на X» / «проверь — уже покупали как X» / «проверь — новая позиция» | Нужен взгляд человека |

**На период обучения — намеренно строго:** даже уверенное AI-совпадение с эталоном
считается жёлтым (не зелёным). Цель — приучить проверять глазами, а не доверять слепо;
ослабить можно позже, когда алиасов накопится больше.

**Счётчик** `#check-counter` наверху — «⚠ N строк требуют проверки» / «✓ всё проверено».
Обновляется на клиенте (JS) при выборе позиции из автокомплита (`selectCandidate` в
`confirm.html`), не только при первой загрузке страницы — сервер знает только начальное
состояние, дальнейшее «я проверил» существует только как client-side flag
(`tr.dataset.checked`), нигде не сохраняется в БД.

**Правило для персонала:** жёлтая рамка = нужен взгляд, сначала смотри на выпадающий
список подсказок (автокомплит, `/expenses/products/search`), а не печатай с нуля.

## Confirm flow (обновлено 06.07 — категория больше не спрашивается)

1. `GET /{id}/confirm` — рендер с позициями из OCR, нормализация через Gemini (сопоставление
   + AI-предложение имени для новых позиций)
2. Пользователь проверяет позиции (цветовые подсказки), сумму, дату, поставщика (обязателен)
3. `POST /{id}/confirm` — резолвит товар по каждой позиции (`get_or_create_product`, точное
   совпадение по тексту в поле «Название» — см. предупреждение ниже), группирует по
   `expense_category_id`, создаёт Transaction(s) с пропорциональным делением суммы/долга,
   WarehouseReceipt если `warehouse_eligible`. Автопромоута в эталон здесь больше нет —
   убран 08.07, см. раздел выше.
4. Redirect → экран успеха (та же страница в confirmed state), кнопка "+ Следующая
   квитанция" — единственная крупная (главный шаг цикла массового ввода), "Непроведённые"/
   "К списку" — мелкие второстепенные ссылки (успех-экран нужен для галочки-подтверждения
   нетехническому персоналу, но не должен тормозить массовый ввод)

**Split:** одна квитанция → несколько Transaction разных категорий через
ReceiptTransaction — теперь автоматически по товару, не ручным выбором.

## Workflow персонала (ежедневный, обновлено 06.07)

```
Фоткает чек → загружает в систему
    ↓
OCR читает позиции (2-5 сек)
    ↓
Страница подтверждения:
  Зелёная рамка → ничего не делает
  Жёлтая рамка → смотрит на выпадающий список подсказок, выбирает или правит
    (для совсем новых позиций — уже готовое AI-предложение имени, не с нуля)
    ↓
Счётчик «N строк требуют проверки» → 0 когда все выбраны/подтверждены
    ↓
Категория НЕ спрашивается — определится сама по товару при проведении
    ↓
"Провести" → "+ Следующая квитанция" (главная кнопка, сразу к следующему чеку)
```

Позиции обязательны для любого товарного чека, включая Ремонт/Хозяйство — каталог
для этих категорий появился 06.07 (раньше не было). Исключение — Транспорт/Прочее
(услуги, не раскладываются на позиции, решение отложено).

## Денежная логика позиций — доведена до строгой (07.07, весь день)

После того как Мунара/Махабат начали реально вводить чеки с новой авторазбивкой
категорий (06.07), всплыла системная дыра: несколько мест в коде всё ещё тихо
доверяли числам напрямую из OCR, а не пересчитывали их из проверенных человеком
полей — ровно то, от чего должна была защищать итемизация. Пять последовательных
фиксов за один день, каждый вскрывал следующий слой той же проблемы:

1. **OCR-промпт расширен** под реальные форматы чеков (двухколоночные списки,
   перенос имени на строку таблицы, повёрнутый/чужой контент на обороте
   переиспользованной бумаги) — устранён баг с цифрой/формулой вместо имени
   товара. Позиция без цены (OCR нашёл только общий итог) больше не теряется
   молча — подсвечивается «впиши цену», блокирует «Провести». Миграция:
   `receipt_items.total_price` стало nullable.
2. **Сервер перестал доверять присланному `item_total_price`** — total всегда
   `qty * unit_price`, и на confirm, и на add. Нет кол-ва/цены → в confirm
   это блокирует проведение, в add (позиции необязательны) — позиция просто
   не сохраняется. Поле суммы в обеих формах стало readonly.
3. **Позиции, которые ИИ сам не считает товаром** (мусор/пометки/цифры без
   уверенности в `clean_name`), больше не попадают на confirm как «новая
   позиция» — раньше это создавало ложное впечатление достоверности.
4. Но простое скрытие таких строк создавало **новую дыру**: их деньги молча
   выпадали, JS пересчитывал «Сумма» только по видимым строкам, и при
   ручном восстановлении верного итога недостача **пропорционально
   размазывалась по реальным категориям** (`scale = amount/items_sum`) —
   портя ровно ту статистику, ради которой затевалась итемизация. Фикс:
   сумма скрытых строк уходит в `group_totals[None]` («без категории»)
   честно, `scale` остаётся ≈1.
5. **Финальный шаг** — даже это оказалось полумерой: перенос суммы в «без
   категории» — это всё ещё вставка цифры из OCR в проводку без проверки
   человеком. Теперь для нераспознанных строк денег не остаётся **вообще
   нигде** — только счётчик «N строк не распознаны, на сумму ~X» и
   нередактируемая ссылка на исходный OCR-итог чека для сверки. Если это
   реальная покупка — человек допишет её сам, глядя на чек. Заодно
   починена та же дыра в отображении «полных» позиций: confirm-страница
   показывала сырой `total_price` из OCR, а не `qty×price` — если человек
   не трогал поля, видимое число могло разойтись с тем, что реально
   сохранится (сервер и так всегда считает сам). Теперь отображение =
   сохранение.

**Правило, вынесенное из этой цепочки:** ни одна денежная цифра в проводке не
должна происходить напрямую из OCR — только из полей, которые прошёл через
проверку человеком (или явно посчитаны из них). Любое отклонение от этого
обнаруживается не сразу, а на следующем слое (размазывание по категориям,
расхождение отображения) — при следующих похожих правках проверять денежный
путь до конца цепочки, не только на первый взгляд очевидное место.

## Три потока расходов: Закуп / Сервис / Ежемесячные (12–13.07)

До 12.07 весь неквитанционный расход шёл через одну форму `/expenses/add`
(«Вручную») с общим дропдауном категорий — включая ФОТ/Охрану/Коммуналку,
что порождало нелепые формы (обязательный «Поставщик» для зарплаты — баг,
найденный и исправленный дважды, второй раз после консолидации категорий
переименовал условие фильтра и снова открыл ту же дыру). Разведено на три
явных потока с разной природой:

| Поток | Экран | Природа |
|---|---|---|
| Закуп | `/expenses/upload` (фото) + `/expenses/add` (обычная категория) | Товар с позициями, авто-категория по товару |
| Сервис | `/expenses/add?category_id=<id Сервисных>` | Разовый нетоварный расход (такси/доставка/мелкий ремонт), без позиций |
| Ежемесячные | `/expenses/recurring` | Регулярный расход по справочнику шаблонов, привязан к периоду |

Левое меню отражает это одним пунктом «Расходы» (не «Закуп» и не
«Ежемесячные» отдельно) — Ежемесячные и Сервис доступны кнопками на
странице `/expenses/`, не отдельными пунктами сайдбара. Причина: отдельный
пункт «Ежемесячные» в сайдбаре был неочевиден сам по себе без контекста
(не ясно, что это подраздел расходов) — решение пользователя 12.07.

### Форма «Сервис» — категория и поставщик зафиксированы

`/expenses/add?category_id=<id>` при `category_id == id категории «Сервисные
расходы»` переключает `add.html` в `is_service_mode`:
- Категория — не select, а disabled-инпут + скрытый `<input type=hidden>` с
  тем же значением. Пользователь не может случайно сменить категорию.
- Поле «Поставщик» убрано целиком (не просто необязательно — его нет в
  форме). Причина: таксисты, курьеры, разовые мелкие подрядчики меняются
  от раза к разу — трекать их как `Supplier` с историей задолженности не
  имеет смысла, это не тот же тип отношений, что с оптовым поставщиком
  продуктов. На бэкенде `supplier_id = NULL` для таких транзакций.
- Комментарий (`description`) стал **обязательным**, и в форме, и на
  сервере — без поставщика и без позиций это единственное поле, которое
  говорит, на что реально потратили. Валидируется в `handle_add()`.

Определение "это сервисная категория" идёт через `get_service_category_id(db)`
(ищет корневую категорию по имени из `SERVICE_CAT_NAMES`), не через отдельный
query-флаг типа `?mode=service` — так сервер не доверяет клиенту в вопросе
"это сервис или обычная покупка", доверяет только выбранной категории.

### Ежемесячные — идемпотентность разная для ФОТ и для всего остального

`RecurringExpenseTemplate.amount_source` определяет не только откуда берётся
подсказка суммы, но и **всю модель правки** после проведения:

| `amount_source` | Повторное проведение в периоде | Исправить проводку | Удалить проводку |
|---|---|---|---|
| `employees_sum` (ФОТ) | Разрешено сколько угодно раз (аванс + остаток) | **Нет** — только через `/employees/` (правка оклада) | **Нет** — та же причина |
| `last_amount` / `manual` (Охрана, Электричество, Вода, Отопление, Интернет) | Заблокировано, пока не удалена текущая | Да, `/expenses/tx/{id}/edit` | Да, `/expenses/tx/{id}/delete` (soft-delete) |

**Почему асимметрия:** для ФОТ источник истины — оклады сотрудников на
`/employees/`, сама Transaction — производная запись. Править сумму
проводки напрямую задним числом создало бы расхождение между тем, что
реально должны сотрудникам (по их окладам), и тем, что записано в
расходах — а следующий месяц продолжит считать подсказку от актуальных
окладов, не заметив ручной правки прошлой проводки. Для остальных
ежемесячных такого второго источника истины нет: счёт на электричество
существует только в виде введённой цифры, значит саму эту цифру и
следует поправить или удалить при ошибке.

Оба правила защищены с двух сторон: скрыты в UI (`recurring.html`,
условие `row.template.amount_source == "employees_sum"`) **и** проверяются
на сервере в `recurring_post()`/`edit_tx_form()`/`handle_edit_tx()`/
`delete_tx()` — прямой POST в обход UI на ФОТ-проводку или на повторное
проведение той же не-ФОТ статьи в периоде редиректит, не создаёт/не
меняет запись молча.

### `Transaction.period` vs `Transaction.date`

`date` — когда реально оплачено. `period` — за какой месяц расход
относится. Разделены, т.к. расход за июль иногда проводят фактически в
начале августа (коммунальные счета приходят с задержкой). Список на
`/expenses/recurring?month=YYYY-MM` фильтрует **по `period`**, не по
`date` — иначе проводка "потерялась" бы в неправильном месяце.

### Консолидация категорий (12.07, дважды)

Зарплата/Охрана/Коммуналка → «Ежемесячные расходы»; Транспорт/Прочее/
Банковские комиссии → «Сервисные расходы». Внутри одной консолидированной
категории конкретный вид расхода различается **описанием проводки**
(комментарием), не отдельной под-категорией — осознанное решение отложить
детализацию до момента, когда станет ясно по факту накопленных данных,
что встречается достаточно часто, чтобы завести под-категорию (принцип
тот же, что и для роста каталога продуктов — не гадать заранее).

**Урок с фильтром категорий:** после переименования/консолидации категории
нужно свериться со всеми местами, которые фильтруют по **старому имени**
строкой (`RECURRING_ONLY_NAMES`, `SERVICE_CAT_NAMES` в `expenses.py`) — иначе
фильтр молча перестаёт работать и старая утечка (категория без места в форме
"Вручную") воспроизводится под новым именем. Обнаружено дважды за одну
сессию (12.07 ФОТ→«Зарплата», затем 13.07 «Зарплата»→«Ежемесячные расходы»).

для «Ежемесячных расходов» 12.07: распустить старые родительские категории
(Питание/Хозяйство/Ремонт), их дети переезжают на уровень выше под новым
общим родителем. Проверено что не осталось проводок сиротой: 1 историческая

09.07) висела прямо на корневой «Ремонт» — перенесена на «Текущий ремонт
имущества» при миграции (сумма небольшая, похоже на текущий, не капитальный —
если ошибка, поправить вручную).

**«Капитальный ремонт» выведен из «Товары» намеренно** — это не операционный
расход (OPEX), а капитализация затрат (CAPEX): увеличивает стоимость
здания, амортизируется годами, а не списывается расходом месяца. Смешение
с товарными закупками исказило бы сумму текущих расходов в отчётах.
Согласуется с решением 09.07 (тогда уже развели Текущий/Капитальный ремонт
именно по этой причине).

**Не все предложенные финэкспертом переименования применены** — три
категории решений:
1. **Косметические, без риска** — применены не глядя: Готовая
   еда→Услуги питания, Питьевая вода→Бутилированная вода, Канцтовары→

   про понятность для персонала — сделано по прямой команде «делай исходя
   из фин. грамотности», взвешивай так же при следующих похожих запросах)
2. **Отклонено:** «Игрушки» → «МБП» (малоценные быстроизнашивающиеся

   абстрактный бухгалтерский термин запутало бы персонал, который покупает

   термин для ТОГО ЖЕ понятия» (переименовать можно) от «более широкая

здание не в собственности), «Административные расходы» (7310, если

одним кликом по строке в списке. Переведено на тот же паттерн, что и
`/students/{id}/edit` с 04.07: блок `#tx-view` (текст, только для чтения) +
кнопка «Изменить» → показывает `#tx-form` (те же поля редактирования, что
были). JS: `toggleBlock(name, editing)`, идентичен уже существующему в
`students/edit.html` — не изобретён заново.

**Удаление проводки переехало внутрь режима редактирования** (была в
списке `/expenses/`, кнопка 🗑 в колонке actions) — теперь только через
«Изменить» → «Удалить проводку» внизу формы. Технически это **отдельный
`<form>`**, не вложенный в форму редактирования (вложенные `<form>` —
невалидный HTML, браузер ломает поведение) — оба `<form>` лежат внутри
общего `<div id="tx-form">`, который целиком скрывается/показывается через
`toggleBlock`.

**Уже подтверждённые квитанции (`confirm.html`, ветка `ocr_status ==
'confirmed'`) раньше не имели вообще никакого действия** — экран
«Проведено!» был чистым read-only без единой кнопки. Добавлена только
«Удалить квитанцию» (переиспользует существующий `delete_receipt`).
Полноценное редактирование позиций/суммы/категории уже подтверждённой
квитанции — сознательно не сделано, отдельная более крупная задача (чек
мог разъехаться на несколько `Transaction` по категориям при авторазбивке).

## Список расходов — два реальных бага найдены и исправлены (13.07)

Оба существовали независимо от изменений этой сессии, всплыли при живом
тестировании:

1. **Кнопка удаления в `list.html` была нерабочей для всех строк без
   квитанции** (`row_type='manual'` — и Сервис, и Ежемесячные). Форма
   удаления жёстко слала `POST /expenses/{{ r.id }}/delete` — роут для
   удаления *квитанции*, а у ручных транзакций `r.id = None` в питоне
   (используется `r.tx_id`, не `r.id`). Реальный запрос уходил как
   `POST /expenses/None/delete`, FastAPI отбивал 422 (путь ждёт `int`).
   Фикс: для `row_type='manual'` кнопка теперь шлёт на
   `/expenses/tx/{tx_id}/delete` (правильный роут).

2. **`delete_receipt()` падал нарушением внешнего ключа**, если у квитанции
   была хотя бы одна проведённая `Transaction` — то есть удаление было
   сломано для *любого подтверждённого чека*, не только новых. Причина:
   код удалял `Transaction` в цикле, пока `ReceiptTransaction` (FK на
   `transaction_id`) по ней ещё не снят. Фикс — поменян порядок: сначала
   `WarehouseReceipt` → потом `ReceiptTransaction` (снимает FK) → потом
   `Transaction` → `ReceiptItem` → файл → `Receipt`. Раньше эта кнопка,
   похоже, ни разу не тестировалась на реально подтверждённом чеке.

**Клик по строке для Ежемесячных проводок** — раньше вёл в общую форму
редактирования (`/expenses/tx/{id}/edit`), где можно было случайно сменить
категорию или объект в обход `/expenses/recurring`. Теперь для строк с
`recurring_template_id` (флаг `is_recurring` в словаре строки) — ведёт на
`/expenses/recurring?month=<period>`, кнопка удаления в списке скрыта
(своя есть на самой странице Ежемесячных).

**Колонка «Объект» убрана из `list.html`** — пилот сейчас работает только
на одном объекте (Садик Сокулук) единовременно на пользователя, колонка
была чистой избыточностью. Взамен — под названием категории мелким текстом
`tx.description` (для Ежемесячных = имя шаблона, для Сервис = обязательный
комментарий) — решает жалобу «непонятно, это расход за что именно»
(коммунальные — за газ или за воду?). **Когда Школа/Кожомкул подключатся и
один пользователь увидит несколько объектов одновременно — колонку нужно
будет вернуть**, держать в уме перед следующим расширением пилота.

## Посещаемость — данные собираются, но никуда не потребляются (13.07, наблюдение)

`Attendance` (present/absent по каждому активному ребёнку, каждый день,
когда форма сохранена — не только по отмеченным отсутствующим, сервер
проставляет `present=True` всем, кого не отметили) используется **только**
в `attendance.py`, больше нигде в коде. Изначальная задумка 09.07 (снять
вопрос «повар готовит на списочный состав или на факт») требовала, чтобы
Меню/Списание читали эту таблицу — этого не сделано. Данные копятся
исправно (полная история по дням через `?date=`), но ни один другой модуль
их не читает. Идея для будущего — месячный вид на карточке ребёнка с
подсветкой пропусков, обязательно связанный с фактическим уменьшением
`Charge` (не просто отчёт) — записана в `context/ideas.md` (13.07), не
начинать без отдельного обсуждения архитектуры.

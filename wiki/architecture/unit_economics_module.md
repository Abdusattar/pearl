# Сотрудники / ФОТ / Амортизация — Архитектура и решения

> Реализовано: 10.07.2026. Blueprint (проектирование ДО кода, полная
> история решений с датами) — [[blueprints/unit_economics_module]].

## Зачем

Полная картина расходов для юнит-экономики требует ФОТ и амортизации, не
только питания/хозтоваров/закупа (см. `project_brief.md`, «Стратегическое
видение — продолжение 08.07»).

## Стратегическое решение — пересмотр Pearl vs 1С

Решение «закрыто 18.06» (Зарплата/Амортизация — целиком зона 1С) частично
пересмотрено: Pearl считает эти величины сам, **управленческой оценкой**
(не официальный бухучёт — НДФЛ/Соцфонд/проводки остаются зоной
бухгалтера/1С-аналога). Причина — не привязывать продукт структурно к
наличию 1С у клиента (цель на будущее — единый софт для садиков, не все
его имеют).

## Модель данных

### Employee
```
id, organization_id, full_name, role, salary,
status (active|terminated), created_by, created_at
```
Статус — отметка, не удаление (тот же паттерн, что `Student`
active/archived) — история не теряется при увольнении. Отпуск отдельного
статуса не требует, зарплата в отпуске всё равно платится.

### Asset.useful_life_months (расширение)
```
+ useful_life_months  Integer, nullable
```
`NULL` = не амортизируется. Предзаполняется по умолчанию из
`DEFAULT_USEFUL_LIFE_MONTHS` (категория → срок), можно поправить вручную.

Миграция: `e5f6g7h8i9j0`.

## ФОТ и Амортизация — НЕ Transaction, считаются на лету

**Важное архитектурное решение, отклонившее исходный план.** Первый вариант
блюпринта предлагал генерировать эти величины как `Transaction` тем же
паттерном, что `generate_monthly_charges()` (идемпотентно раз в месяц).
Отклонено после разбора с эдвайзером:

- Список детей для `Charge` уже заполнен на момент первого захода — паттерн
  безопасен. Список сотрудников стартует **пустым** — тот же паттерн
  заморозил бы ФОТ первого месяца на 0 навсегда (идемпотентность не
  позволила бы перегенерировать после того, как сотрудников добавили)
- Амортизация — не «текущая сумма», а ряд по времени (стартует через месяц
  после покупки, заканчивается когда актив самортизирован) — не подходит
  под «сгенерировать раз в месяц»
- Синтетические `Transaction` без поставщика/чека исказили бы счётчики
  долга поставщику и «без категории», рассчитанные на реальные закупки

**Решение:** обе величины — чистые функции (`app/services/unit_economics.py`),
считаются заново при каждом обращении, нигде не хранятся:

```python
monthly_payroll(db, org_id)       # SUM(Employee.salary WHERE status=active)
monthly_amortization(db, org_id)  # SUM по активам: cost/useful_life_months,
                                   # с обрезкой по накопленной сумме
```

Показываются на `/employees/` и `/assets/` (итог сверху списка), в
`/expenses/`/Закуп не попадают — это не покупки. Категории расходов
«Зарплата»/«Амортизация» не заводились — не нужны.

**Проверено эмпирически (10.07):** добавление сотрудника → ФОТ вырос;
увольнение → ФОТ сразу упал до 0 (не «застрял» бы, как случилось бы с
Transaction-подходом). Актив с датой покупки 6 месяцев назад,
60 000 сом / 60 мес. → амортизация 1 000 сом/мес., совпало с расчётом
вручную.

## Доступ — роль `founder`

Новая роль, не переиспользует `role=owner` (тот — технический доступ
Абдусаттара, `seed.py` id=1). Заведена для **Айдай** (реальный
бизнес-владелец) — `scripts/add_user.py`, пароль `004` (по аналогии с
000/001/002/003).

- `get_accessible_orgs()` — `founder` в одной группе с `owner`/`director`,
  полный доступ ко всем организациям
- `/employees/` — доступ только `owner`/`founder`. Не Мунаре/Махабат/Айжан
  — оклады чувствительнее, чем остальные данные, к которым у них уже есть
  доступ. Проверено: Айжан (`role=director`) редиректится с `/employees/`

## Экраны

- **`/employees/`** — итог «ФОТ в этом месяце» сверху, форма добавления,
  список (ФИО/роль/оклад/статус). Карточка `/employees/{id}/edit` — тот же
  паттерн просмотр→«Изменить»→«Сохранить» с подтверждением, что на карточке
  ребёнка (нетехнический пользователь не должен бояться случайно всё
  сломать)
- **`/assets/`** — расширена: поле «Срок службы» с плейсхолдером по
  умолчанию (JS меняется при выборе категории), колонка «Амортизация/мес.»
  в таблице, итог «Амортизация в этом месяце» сверху

## Не в V1

- НДФЛ/Соцфонд/удержания — никогда не в Pearl
- История изменений оклада — не версионируется, ФОТ «на сегодня» точный, ФОТ
  «на дату в прошлом» посчитать нельзя (нет истории)
- Тарифный лимит на вычет рекламных расходов по НК КР — отдельный вопрос
  финэксперту, не блокирует

## Новые статьи расходов (отдельно от ФОТ/Амортизации)

Коммуналка (+ Электричество/Вода/Отопление/Интернет), Охрана, Реклама —
решены 10.07, ещё не заведены как `ExpenseCategory` (чистая миграция без
кода). Аренда — не нужна, помещение в собственности.

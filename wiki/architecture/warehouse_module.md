# Складской модуль — Архитектура и решения

> Реализован: 18.06.2026, обновлено 09.07.2026 (гап-анализ + актуализация)

## Зачем Pearl ведёт склад (не 1С)

Повар и завхоз физически принимают товар и списывают продукты. Они не работают в 1С. Pearl даёт им простой интерфейс: приход → кнопка, списание → кнопка. Бухгалтер получает итоговый акт раз в месяц.

## Модели данных

### Product (расширен)
```python
unit      = String(10)   # кг, л, шт, г, уп, пач
category  = String(50)   # мясо, молочные, крупы, овощи, фрукты, прочее
```

### WarehouseReceipt (приходная накладная)
```
date, product_id, quantity, price_per_unit, total_cost
organization_id, supplier_name, transaction_id (→ OCR расход)
```

### WriteOff (акт списания / меню-требование)
```
date, product_id, quantity, organization_id
children_count  ← поле в БД осталось, но убрано из формы 09.07 (см. ниже)
reason          ← питание детей / порча / инвентаризация
```

**09.07: `children_count` убран из формы списания.** Поле ничем не было
подкреплено — вписывалось на глаз, без источника данных (посещаемости нет).
Столбец в БД остался (старые записи не трогали), но новые списания его не
заполняют. Вернётся, когда появится реальный источник числа — либо утренняя
отметка посещаемости, либо содержимое меню. См. гап-анализ ниже.

## Расчёт остатков

```
balance = SUM(WarehouseReceipt.quantity) - SUM(WriteOff.quantity)
          WHERE organization_id IN (current_org + descendants)
          AND deleted_at IS NULL
```

Средняя взвешенная цена: `SUM(total_cost) / SUM(quantity)` из приходов.

## Юнит-экономика (цель)

`children_count` на каждом списании позволяет считать:
```
себестоимость питания на ребёнка в день =
  SUM(write_offs.quantity × avg_price) / SUM(write_offs.children_count)
```
Это KPI для собственников: сколько стоит накормить 1 ребёнка в каждой локации.

## Меню → списание: прогноз vs факт (уточнено 08.07)

Развитие идеи из раздела «Юнит-экономика» выше — не заранее заданный рецепт
блюда с фиксированными граммовками, а сравнение **ожидания** и **факта**:

- Меню на неделю + известное число детей (`children_count`, поле уже есть в
  `WriteOff`) → система прикидывает ожидаемый расход продуктов
- Повар при готовке пишет фактический расход по каждому приёму пищи отдельно
  (завтрак/обед — раздельные `WriteOff` с `reason='питание детей'`)
- Расхождение ожидание↔факт — просто подсвечивается, не блокирует
  проведение. Со временем сами фактические записи формируют реальную
  рецептуру блюд снизу вверх, а не потому что кто-то заранее её вписал

Та же логика (норма по периодичности vs факт) переносима и на хозтовары
(`reason` вне питания). Полный контекст решения — `context/project_brief.md`,
раздел «Стратегическое видение — продолжение (08.07)». Это ещё не
реализовано — уточнение направления перед реализацией большого бэклог-пункта
«Меню → Склад».

**Конкретика для будущей модели данных (уточнено 08.07):** `WriteOff` в
текущем виде привязан только к `reason` (категория) — этого недостаточно,
нужна привязка к конкретному приёму пищи (условно `meal_id` или
`(date, meal_type)`), не просто «питание детей» общей строкой. Меню на
неделю берётся заранее и служит источником для обеих сторон сравнения:
и ожидания (примерная раскладка по блюду), и факта (сверяется с тем, что
повар реально списал на этот приём пищи).

### Два разных блокера — важно не путать

**Корзина A — можно строить сейчас, ни от чьего решения не зависит:**
Абдусаттар лично знает содержание меню и может его продиктовать, даже если
в системе нет UI для его ввода собственника/повара. Сюда входит:
- Сущность меню/блюда (даже в простом виде — список блюд + к какому приёму
  пищи относится)
- Привязка `WriteOff` к конкретному блюду и приёму пищи (`meal_type` +
  `dish_id` или похожее) — закрывает прямые вопросы «почему нет завтрака
  в причине» и «за какое блюдо списание»

**Корзина B — упирается в ответ Мунары/Айжан по посещаемости, не в разработку:**
- Ожидаемый расход (прогноз) — масштабируется на число детей
- Себестоимость на ребёнка
- Расхождение ожидание vs факт — «ожидание» само зависит от числа детей
- Смысл самой отметки посещаемости (влияет ли на скидку родителям, готовит
  ли повар на списочный состав или на факт) — это решение Мунары/Айжан,
  не то, что можно додумать самим

Сама механика записи «кто сегодня отсутствует» технически не зависит от
ответа — можно построить независимо (см. обсуждение чекина по группам в
`logs/2026-07-09.md`). Но результат этой записи пока некуда применить по
существу (скидка? объём готовки?), пока нет ответа.

**Уточнение того же дня (09.07, позже в сессии):** Абдусаттар снял ровно
половину этого блокера — **механизм**, не политику. Утренняя отметка
(~10:00, по группам, чекбокс «отсутствует») даёт факт числа детей **до**
готовки — значит вопрос «повар готовит на списочный состав или на факт»
решается сам собой: на факт, число известно заранее, отдельного прогноза
для количества детей не нужно (прогноз и факт здесь — одно и то же число).
Это разблокирует расчёт себестоимости на ребёнка, как только чекин +
привязка списания к приёму пищи существуют.

**Вопрос скидки родителям за дни отсутствия — решён 10.07.** Это биллинг
(`Charge`/баланс ребёнка), не склад/меню — ответ Мунары: минус 100 сом за
день пропуска. Реализация ещё не спроектирована, детали —
[[architecture/billing_module]]. Не путать эти два вопроса — оба сняты, но
это два независимых решения, а не одно.

**V1 сознательно без прогноза/нормы по продуктам.** Отдельная норма
«ожидание vs факт по количеству продукта на блюдо» тоже не строится в первой
версии — по видению 08.07 рецептура должна всплыть снизу вверх из
накопленных фактических записей, сравнивать не с чем, пока не накопится
история за несколько недель.

**Что реально нужно для V1 (после этого уточнения):**
1. Контент меню — придёт бумагой/Excel от Мунары/кухни, Абдусаттар сам
   вносит на первое время (UI для собственника — позже, не в V1)
2. Сущность блюда/приёма пищи + привязка списания к ним
3. Утренний чекин посещаемости по группам (детская явка на день)
4. Себестоимость на ребёнка — считается напрямую из (2)+(3), без нормы/прогноза

Скидка за отсутствие остаётся открытым вопросом в отдельном (биллинговом)
модуле, к этому списку не относится.

## Авто-создание прихода из квитанции (18.06, поведение уточнено 09.07)

При подтверждении квитанции:
- Показывается зелёный чекбокс «Добавить в склад», если хоть одна позиция с количеством
- При Провести → `WarehouseReceipt` создаётся для каждой позиции с qty > 0 и price > 0
- `WarehouseReceipt.transaction_id` = id созданной транзакции расхода

**Уточнение 09.07:** чекбокс НЕ смотрит на `warehouse_eligible` (флаг мёртвый,
см. выше) — показывается для любой категории с количеством, включая
непищевые. На практике этим пока пользовались только для еды.

## Цветовая индикация остатков

- Красный: `balance < 1` (критический минимум)
- Жёлтый: `1 ≤ balance < 3`
- Нормальный: `balance ≥ 3`

автоматически ставят товар на склад через галочку «Добавить в склад». Это
не затронуто сегодняшней консолидацией — отдельный, основной путь.

## Актуализация: загрузка текущих остатков (09.07, вторая волна)

Кнопка «⤓ Загрузить все текущие остатки» на `/warehouse/actualize` —
альтернатива ручному вводу по одному товару через автокомплит. Не заменяет
его, а дополняет: подходит для полной инвентаризации (пересчитал весь склад
физически), тогда как автокомплит остаётся для точечных правок.

- По клику таблица заполняется одной строкой на каждый товар с ненулевым
  остатком по системе, `item_product_id` проставляется сразу (без нечёткого
  поиска при сохранении — критично, иначе `get_or_create_product` мог бы
  привязать строку не к тому товару или создать дубль)
- Строки группируются заголовками по `Product.category`, сортировка
  `category NULLS LAST, name` — раньше сортировка шла только по имени, из-за
  чего товары разных категорий шли вперемешку
- Изменённая вручную строка подсвечивается (жёлтый фон) — сравнение текущего
  значения с тем, что было загружено изначально (`data-original`), не с БД
  напрямую
- **Крестик ≠ обнулить остаток.** Удаление строки перед сохранением просто
  исключает товар из отправки — сервер не увидит эту позицию вообще, остаток
  в БД не изменится. Чтобы показать «товара больше нет» — нужно вписать 0
  (тогда дельта = -остаток, создаётся списание на всю сумму). Это неочевидно
  и явно прописано в подсказке над формой.

**Бэклог: отчёт по корректировкам актуализации.** В норме на складе, который
ведётся аккуратно, актуализация не должна почти ничего находить — регулярные
крупные подгонки по одному товару сигнализируют, что раньше был неверно
внесён приход/списание, либо инвентаризацию делают слишком редко. Строить
можно уже сейчас без миграций — каждая корректировка уже помечена отдельно
(`WriteOff.reason = "инвентаризация"`, `WarehouseReceipt.supplier_name =
"Инвентаризация"`), не смешана с обычными операциями. Не реализовано,
отложено на потом.

## Переименование категории «Бытовая химия» → «Мыломойка» (09.07)

Миграция `x2y3z4a5b6c7`. Только `ExpenseCategory.name` (id=5) — на момент
переименования ни один товар не был привязан к этой статье расходов
(`expense_category_id = 5` → 0 строк), так что видимого эффекта в интерфейсе
пока нет, появится когда хозтовары начнут заводить с этой категорией.

Важный контекст: 6 июля миграция `o3p4q5r6s7t8` уже **удалила** отдельную
корневую категорию «Мыломойка» как дубликат «Хозяйство → Бытовая химия».
Сегодняшнее переименование — не откат того решения, а просто более понятное
имя для той же (единственной) подкатегории; повторного дубля не возникает.

# OCR — история выбора решения

> Обновлено: 2026-07-05

## Задача

Распознавание фото квитанций (хозяйственные расходы школы и садика):
- Рукописные квитанции (большинство)
- Печатные чеки
- Фото с телефона (разное освещение, качество)
- 92 накопленных фото у Мунары в media/receipts/

## Эволюция выбора

### 1. Google Cloud Vision (первоначальный план, ~май 2026)
- 1000 запросов/мес бесплатно
- Отклонён: непонятна стоимость сверх лимита, привязка к Google Cloud

### 2. EasyOCR (локальный)
- Попытка: локальное решение без API
- Отклонён: качество на рукописях неприемлемое, тяжёлый для VPS

### 3. OpenRouter Vision — qwen/qwen2.5-vl-7b-instruct:free (история)
- Бесплатно, работает через OpenRouter API, неплохо на печатных
- Минус: **на рукописях не справляется** — критично для наших квитанций

### 4. OpenRouter Vision — google/gemini-2.5-flash-lite (текущий, с 19.06)
- **Статус: в продакшне** — `app/services/ocr.py` и `app/services/normalize.py`
- $0.10 / $0.40 за 1M токенов (вход/выход), 1M контекст
- На рукописях справляется заметно лучше, чем qwen2.5-vl-7b:free

## Тест 05.07 — китайские vision-модели как альтернатива

Повод: жалоба на скорость прода потянула за собой вопрос «а нет ли модели
сопоставимой по цене, но лучше именно на рукописных чеках». Прогнали на двух
реальных чеках (один чёткий, один почти нечитаемый даже для человека):

| Модель | Цена вх/вых за 1M | Итог теста |
|---|---|---|
| **google/gemini-2.5-flash-lite** (текущая) | $0.10 / $0.40 | На чётком чеке — 100% верно (все позиции, количества, итог). На грязном — нестабильно (в одном прогоне частичный null, в другом галлюцинация числа) |
| qwen/qwen3-vl-32b-instruct | $0.104 / $0.416 (сопоставимо) | На чётком чеке хуже: исказил названия («керточина» вместо картошка), ошибся в qty (48 вместо 78). На грязном — числа правдоподобнее Gemini, но названия тоже не разобрал |
| z-ai/glm-4.5v | $0.60 / $1.80 (дороже в 4-6 раз) | Провал: на грязном чеке пустой ответ (`content: None`), на чётком — ответ обрезался на `max_tokens=2048` посреди JSON (модель «thinking»-типа, тратит бюджет на рассуждение до финального ответа) |

**Решение: остаться на Gemini 2.5 Flash Lite.** Ни один кандидат не показал
превосходства ни по цене, ни по качеству на этих образцах. GLM-4.5V для
честного теста потребовал бы больший `max_tokens` (и тогда реальная цена
запроса выше заявленной). Скрипт сравнения — в scratchpad сессии, не в
репозитории; при повторном тесте: та же image_url структура запроса, что и
в `ocr.py`, просто менять `model` в payload.

**Вывод про методологию:** 2 чека — маленькая выборка, температура 0.1 даёт
разброс между прогонами одной и той же модели. Если будет ещё повод менять
модель — тестировать на 15-20 чеках с посчитанным % расхождений, не на паре
примеров.

## Архитектура OCR пайплайна

```
Фото → OpenRouter Vision API → JSON позиций → receipt_items в БД → confirm страница (Мунара правит)
```

Мунара **подтверждает** каждую квитанцию на confirm-странице перед сохранением.
Это страхует от ошибок OCR — человек в контуре.

## Что реализовано

- Upload страница с OCR overlay (индикатор прогресса)
- Confirm страница: редактирование позиций, пересчёт qty→итог
- file_hash дедупликация (повторная загрузка того же фото не создаёт дубль)
- Ручная запись без квитанции (если фото нет)
